How to disable and enable malware protection átop

Please note that the ASI configuration steps below will let you easily disable and enable the ASI malware protection Scrips ( 27 System Start-up Environment Control, and 232 – Intrusion Protection Control). You should do this in order to avoid potential Scrip 27 and 232 interference with the installation or upgrade of software applications, or changes to items such as MS Internet Explorer start pages. You will need to disable Scrips 27 and 232 only if you have previously configured them to protect systems at your site (as described in the HFN malware removal instructions.htm document).

If you did not configure Scrips 27 and 232 to protect systems at your site, you do not need to disable them when installing upgrading software applications.

Please note that enabling Scrips 27 and 232 will not automatically enable malware protection. To do that, you will need to follow the instructions in the HFN asi malware removal instructions.html document.

When Scrips 27 and 232 are configured to protect a system against malware, stopping the ASI client does stops malware protection (Scrips 27 and 232), but it will not work as a way to let you make changes to the areas protected by Scrips 27 and 232. When you re-start the ASI client after making any changes to the areas protected by Scrips 27 and 232, the Scrips use the snapshots of the environment they took before the ASI client was disabled to check the system areas they protect. As these snapshots do not include the changes you made while the ASI client was disabled, Scrip 27 and 232 will disable them.

Disabling Scrips 27 and 232, making the desired changes, and then re-enabling Scrips 27 and 232 will cause the Scrips to take new snapshots of the areas they protect as soon as they are enabled. In this way, any changes made to areas protected by Scrips 27 and 232 while they were disabled will not be disallowed.

Please remember that if you disable malware protection (Scrips 27 and 232) via the ASI server, before making any changes on the system(s) where you disabled the Scrips without interference from Scrips 27 and 232, you will need to wait until the disable action has been retrieved by the ASI client on the system(s) in question, and applied locally on the system(s). As you know, the timing of this action depends on frequency of execution of Scrip 177 (Scrip Configuration Update). Please refer to section How to change the frequency with which the ASI client retrieves Scrip configuration changes from the ASI server of this document for step-by-step instructions for changing the Scrip 177 execution schedule.

Method A - ASI server (to be used if you are not on site)

2) Click on the "Use extended interface" link on the right side of the ASI Dashboard upper pane, below user login and date. The “Welcome” page will be displayed.

5) Follow the on-screen instructions

How to enable resource utilization monitoring átop

Please note that by default, Scrip 6 (Memory Statistics), Scrip 96 (Processor Statistics), and Scrip 98 (Network Statistics), the Scrips used to monitor resources utilization, are enabled by default with a configuration already well suited to alert you about unusual levels of resource utilization.

The goal of example below, where you will find instructions for modifying the configuration of Scrip 96 (Processor Statistics), is to show you how you can change the configuration of a resource monitoring Scrip. You can replace the processor resource monitoring thresholds used in the example with others which you believe to be better suited to your requirements.

Method A - ASI server (to be used if you are not on site)

1) Log onto the ASI server (https/asiservername/main ). The ASI dashboard will be displayed.

2) Click on the "Use extended interface" link on the right side of the ASI Dashboard upper pane, below user login and date. The “Welcome” page will be displayed.

3) At the upper right-hand corner of any page on the ASI server, click on the configuration link in the sites navigation bar (https://asiservername/main/config/index.php?act=site ). The “select a Site” page will be displayed.

4) Click on the view machines link for the site of the system where you want to monitor processor statistics. The "Configuration - <site name> Machines" page will be displayed.

5) Click on the configure Scrips link for the system where you want to monitor processor statistics

6) Scroll down to the entry for Scrip 96 (Processor Statistics), and click on the Scrip name

7) If you want to enable Scrip 96 only on the system where you want to monitor processor statistics, select machine <name> from the Group pull-down list for the Scrip enabled configuration parameter. If you want to enable Scrip 96 on all systems at the site, select site <name> from the Group pull-down list for the Scrip enabled configuration parameter is on.

8) Click in the Scrip enabled box

9) We suggest you use the following site <name> group settings:

- Processor queue length above which to report - 10 (Please note that 2-5 is considered normal)

- Processor interrupts per second above which to report - 5000 (This is highly dependent on the processor, and on the applications running on the system)

- Percent processor usage above which to report - 75 (This is subjective)

- Sampling period in milliseconds - 15000 (leave the default unchanged)

10) If you are troubleshooting a problem enable the Report running processes configuration parameter on the system(s) you are troubleshooting. In general, you should minimize the number of systems from which the complete list of processes running on them is posted on the ASI server to minimize bandwidth, and mass storage consumption.

11) Configure Scrip 96 to run once per minute (enter a * in the minute field in the Schedule configuration parameter), and set the Random minute interval to 0.

12) Scroll down to the bottom of the Scrip 96 configuration page, and click on the Submit Changes button. The Changes Recorded page will be displayed.

Please note that you can enable memory utilization monitoring by following the above instructions for Scrip 6 (Memory Statistics), and Scrip 98 (Network Statistics) instead of Scrip 96. The key Scrip 6 configuration parameter is Number of page reads per second above which to report which we suggest you set to 300.

Method B - Direct ASI client access (can be used if you are on site)

1) Open a browser window.

2) In the address box, enter http://localhost:2721/

3) In the user logon dialog enter <direct access user id> / <direct access password>

4) Click on the name of the system at the very top of the page. The Scrip Configurator page will be displayed.

5) Scroll down to the entry for Scrip 96 (Processor Statistics), and click on the Scrip name

6) If you want to enable Scrip 96 only on the system where you want to monitor processor statistics, select machine <name> from the Group pull-down list for the Scrip enabled configuration parameter. If you want to enable Scrip 96 on all systems at the site, select site <name> from the Group pull-down list for the Scrip enabled configuration parameter is on.

7) Click in the Scrip enabled box

8) We suggest you use the following global settings:

- Processor queue length above which to report - 10 (Please note that 2-5 is considered normal)

- Processor interrupts per second above which to report - 5000 (This is highly dependent on the processor, and on the applications running on the system)

- Percent processor usage above which to report - 75 (This is subjective)

- Sampling period in milliseconds - 15000 (leave the default unchanged)

9) If you are troubleshooting a problem enable the Report running processes configuration parameter ONLY on the system(s) you are troubleshooting. It is very important that you minimize the number of systems from which the complete list of processes running on them is posted on the ASI server to minimize bandwidth, and mass storage consumption.

10) Configure Scrip 96 to run once per minute (enter a * in the minute field in the Schedule configuration parameter), and set the Random minute interval to 0.

11) Click on the Submit Changes button. The Changes Recorded page will be displayed.

15) From the Scrip number pull-down list select Scrip 93

16) Click on the Search button

How to run a program, batch file, script, or MS-DOS command remotely átop

1) Log onto the ASI server (https/asiservername/main ). The ASI dashboard will be displayed.

2) Click on the "Use extended interface" link on the right side of the ASI Dashboard upper pane, below user login and date. The “Welcome” page will be displayed.

3) At the upper right-hand corner of any page on the ASI server, click on the configuration link in the sites navigation bar (https://asiservername/main/config/index.php?act=site ). The “Select a Site” page will be displayed.

4) Click on the view machines link for the site of the system where you want to run the executable. The "Configuration - <site name> Machines" page will be displayed. Please note that if you want to run an executable on all systems at a site, you can pick any system at the site.

5) Click on the configure Scrips link for the system where you want to run the executable

6) Scroll down to the entry for one of Scrips 89, 151-157, or 212-215 (Scheduled Program Execution 1, or Scheduled Program Execution 2-5, or Scheduled Program Execution 7-12), and click on the Scrip name. The Scrip configuration page will be displayed.

7) If you want to run the command line once, do not enable the Scrip, otherwise click in the Scrip enabled box. If you want to run the command line only on the system whose Scrip 89 configuration page you are viewing, select machine <name> from the Group pull-down list for the Scrip enabled configuration parameter. If you want to run the command line on all systems at the site, select site <name> from the Group pull-down list for the Scrip enabled configuration parameter.

8) If the executable you want to run resides on a networked resource, or needs to access a networked resource, click on the Impersonation enabled box (you can leave Group selection for the Impersonation enabled configuration parameter set to site <name>). Please refer to the Scrip 89 (151-157, and 212-215) configuration help file for detailed information and configuration instructions for enabling impersonation support (https://asiservername/main/config/help/s00089.htm ).

9) If you want to run the program once, ignore the schedule parameter, otherwise set the schedule as required. For example, if you want to run an executable on Wednesdays at 5.30 AM, you would set the Minute field (the first field from the top) to 30, the Hour field (the second field) to 5, and the Weekday field (the fifth field) to 3, skipping the Day and Month fields. In all likelihood, you would leave the Random minute field (the sixth field) to 0. You would probably also leave the Type field (the seventh field) set to 2, unless you wanted the executable to run immediately when the system starts-up in case the system was shut down when the executable was scheduled to run (at 5.30 AM on Wednesdays). If you left the value of the Fail limit field (the eight field from the top) set to 0, and the Type field set to 2, Scrip 89 (or one of 151-157, or 212-215) would post an event log reporting failure to run because the system was turned off, or the ASI client was not running, the first time such a failure occurs.

10) In the Executable configuration parameter, enter the complete command line for the program, MS-DOS command, batch file, or script you want to run. Please note that you can enter only one command line. If you want to run the command line only on the system where you are configuring the program execution Scrip, select machine <name> from the Group pull-down list. Otherwise, select site <name>.

11) If you want to run the program once, Click on the Run process now Execute button taking care to select machine <name> from the pull-down list to the right of the Execute button if you want to run the executable only on the system whose Scrip 89 (or one of 151-157, or 212-215) configuration page you are viewing.

12) If you configured Scrip 89 (or one of 151-157, or 212-215) to run periodically, Click on the Submit Changes button. The Changes Recorded page will be displayed.

13) Click on the machines link in the second-level navigation bar (the one labeled configuration). The Configuration - <site name> Machines page will be displayed.

14) The entry in the Changed column for the system you configured Scrip 89 (or one of 151-157, or 212-215) to run the executable on will show the date and time of the action you just took. Before being able to retrieve the Scrip 89 event log reporting completion of the executable processing, you will need to wait until the entry in the Changed column changes to No (Use the refresh function in your browser). This is dependent on the frequency of execution of Scrip 177 (Scrip Configuration Update). Please refer to the How to change the frequency with which the ASI client retrieves Scrip configuration changes from the ASI server section of this document for instructions on how to change the Scrip 177 execution schedule.

15) As soon as the entry in the Changed column for the system you retrieved the list of running processes from changes to No, click on the ad-hoc link in the events navigation bar found at the upper right-hand corner of any page on the ASI server (https://asiservername/main/event/event.php ).

In the Select Additional Search Criteria for Events section of the Ad-hoc Event Query Form page:

16) From the Site pull-down list select the site of the system you retrieved the list of running processes from

17) From the Machine pull-down list select the system you retrieved the list of running processes from

In the Narrow Your Search (optional) section of the Ad-hoc Event Query Form page:

18) From the Scrip number pull-down list select Scrip 89 (or one of 151-157, or 212-215)

19) Click on the Search button

20) In the Event Query Results page click on the detail link for the Scrip 89 (or one of 151-157, or 212-215) event log retrieved by the event query you just ran.

Please note that when executing a command line that uses a file (including its path) as a command line parameter, the file name and path should be enclosed between double quotes (“). For example,

regedit /s "c:\DefwatchQSOff.reg"

You should also note that at this point in time, completion status information reported by Scrip 89 event logs is limited to reporting successful or failed execution. If you want to get more information about the execution and the executable you just ran remotely can produce an output or log file, you can use Scrip 100 to retrieve such a file. Please refer to section How to retrieve files remotely in this document.

How to retrieve files remotely átop

Scrip 100 is a powerful and highly configurable Scrip. Below, you will find instructions for getting to the Scrip 100 configuration page and guiding you in the configuration Scrip to retrieve a file. These instructions do not include detailed information about each Scrip 100 configuration parameter The instructions do not include detailed information about each Scrip 100 configuration parameter. Please also note that we have comprehensive guides for the configuration of Scrip 100 to retrieve log files produced by a variety of applications including Veritas Backup Exec, AVG anti virus, and Spybot. 12) Next, you can configure Scrip 100 to retrieve files based on their last modified date. (MM/DD or MM/DD/YY, leave blank if you want current date) configuration parameter. You can also use the How many days back from the latest date do you want to retrieve files for? configuration parameter to limit the number of files retrieved. For example if you set the How many days back from the latest date do you want to retrieve files for? Configuration parameter to 0, Scrip 100 will only retrieve files whose last modified date match the date entered in the What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date) configuration parameter.

13) In the File distribution and retrieval command line(s) configuration parameter, you combine all of the information entered in the other Scrip 100 configuration parameters into the command line(s) Scrip 100 will use to retrieve files for you. You can enter as many file retrieval command lines as you like, enabling and disabling them as needed. Please refer to the Scrip 100 configuration help file for detailed information on file distribution and retrieval command line format.

Please note that In the File distribution and retrieval command line(s) configuration parameter, you can enter both file retrieval and distribution command lines, in any order. Scrip 100 will automatically detect whether a command line is for retrieving or distributing a file based on the contents of the command line, e.g. whether the command line contains an e-mail group, or a file location group entry.

14) If the file retrieval you want Scrip 100 to perform is a one-time operation, ignore the schedule parameter. Otherwise set the schedule as required. For example, if you want to retrieve a file every day at 7 AM, you would set the Minute field (the first field from the top) to 0, the Hour field (the second field) to 7, and the Weekday field (the fifth field) to *, skipping the Day and Month fields. In all likelihood, you would leave the Random minute field (the sixth field) to 0. You would probably also leave the Type field (the seventh field) set to 2, unless you wanted the file retrieval operation to occur immediately when the system starts-up in case the system was shut down when the file retrieval was scheduled to occur (at 7 AM every day). If you left the value of the Fail limit field (the eight field from the top) set to 0, and the Type field set to 2, Scrip 100 would post an event log reporting failure to run because the system was turned off, or the ASI client was not running, the first time such a failure occurs.

If you want to the Scrip 100 schedule to apply only to the system whose Scrip 100 configuration page you are viewing, select machine <name> from the Group pull-down list for the Scrip enabled configuration parameter. If you want to distribute one ore more files to all systems at the site, select site <name> from the Group pull-down list for the schedule configuration parameter.

15) If you configured Scrip 100 to upload the file(s) it retrieves to your ASI server, for ease of processing, transfer, and taking into account the default 64 KB limit on event log size (configurable), you may want to have the Scrip create a separate event log for each file retrieved. If you want to do this, enable the the Log each file in separate event logs configuration parameter.

16) Click on the Distribute / Retrieve file(s) now Execute button taking care to select machine <name> from the pull-down list to the right of the Execute button if you want to run the file(s) only on the system whose Scrip 100 configuration page you are viewing.

17) If you configured Scrip 100 to perform a periodic file retrieval operation, click on the Submit Changes button. The Changes Recorded page will be displayed.

18) Click on the machines link in the second-level navigation bar (the one labeled configuration). The Configuration - <site name> Machines page will be displayed.

Before being able to retrieve the Scrip 100 event log reporting completion of the file retrieval action, you will need to wait until the entry in the Changed column changes to No (Use the refresh function in your browser). This is dependent on the frequency of execution of Scrip 177 (Scrip Configuration Update). Please refer to the How to change the frequency with which the ASI client retrieves Scrip configuration changes from the ASI server section of this document for instructions on how to change the Scrip 177 execution schedule.

20) As soon as the entry in the Changed column for the system you configured Scrip 100 on changes to No, click on the ad-hoc link in the events navigation bar found at the upper right-hand corner of any page on the ASI server (https://asiservername/main/event/event.php ). How to stop, re-start processes and services átop

From time to time, you may want to stop running services or processes which you suspect may be causing performance problems on one or more systems, or you may want to stop execution of suspected malware processes.

To perform this operation remotely on one, some, or all systems at a site, without having to take remote control of these systems, you can use Scrip 227 as described below. Please note that Scrip 227 will not start stopped processes or services. You can use Scrip 176 to start stopped services, and one of Scrips 89, and 212-215 to start stopped processes (please refer to section How to run a program, batch file, script, or MS-DOS command remotely of this document).

Scrip 227 is a powerful and highly configurable Scrip. Below, you will find instructions for getting to the Scrip 227 configuration page and guiding you to the relevant configuration parameters, depending on what you want to do. The instructions do not include detailed information about each Scrip 227 configuration parameter. Clicking on the help link on the second-level navigation bar (labeled configuration on the right side of the Scrip 227 configuration page) will open the Scrip 227 configuration help file in a new window on your desktop (https://asiservername/main/config/help/s00227.htm ).

1) Log onto the ASI server (https/asiservername/main ). The ASI dashboard will be displayed.

2) Click on the "Use extended interface" link on the right side of the ASI Dashboard upper pane, below user login and date. The “Welcome” page will be displayed.

3) At the upper right-hand corner of any page on the ASI server, click on the configuration link in the sites navigation bar (https://asiservername/main/config/index.php?act=site ). The “Select a Site” page will be displayed.

4) Click on the view machines link for the site of the system where you want to stop execution of services and/or processes. The "Configuration - <site name> Machines" page will be displayed. Please note that if you want to stop and/or re-start processes and services on all systems at a site, you can pick any system at the site.

5) Click on the configure Scrips link for the system you want to stop and/or re-start processes / services on.

6) Scroll down to the entry for Scrip 227 (Process and Service Shutdown-Restart), and click on the Scrip name

7) If you want to stop or re-start processes and/or services once, do not enable the Scrip, otherwise click in the Scrip enabled box. If you want to stop or re-start processes and/or services only on the system whose Scrip 227 configuration page you are viewing, select machine <name> from the Group pull-down list for the Scrip enabled configuration parameter. If you want to stop or re-start processes and/or services on all systems at the site, select site <name> from the Group pull-down list for the Scrip enabled configuration parameter is on.

8) Enter the process(es) and/or services(es) you want to stop or re-start in the Process groups and Services groups configuration parameters respectively, following the instructions in the Scrip 227 configuration help file (https://asiservername/main/config/help/s00227.htm ).

Unless you have a very clear reason to do otherwise, you should leave the Process groups and Services groups configuration parameters Group parameter pull-down list selection set to site <name>. In this way, you will have to enter information about processes and services which you might want to stop or re-start at some point on any system at your sites, only once. For example, suppose you want to stop a malware process which you found running on a system (using the instruction in the How to find out remotely which processes are running on a system section of this document). You can use Scrip 227 to do that. By leaving the Group parameter pull-down list selection set to site <name>, Scrip 227 on every other system at the site will be automatically configured with the same malware process group information, and using the export function you can export the Scrip 227 global configuration parameters to all of your sites.

9) Initially, you can leave the Max number of seconds to wait for processes to close, and Max number of dialogs to close for a process configuration parameters unchanged.

10) If you are performing a one-time process / service shutdown / re-start operation, enter the names of the process and/or services groups in the On demand shutdown and/or On demand restart respectively, and click on the Process on demand items now Execute button. The Changes Recorded page will be displayed.

11) If you want to configure Scrip 227 to perform a periodic process / service shutdown / re-start operation, enter the names of the process and/or services groups in the Scheduled shutdown and/or Scheduled restart respectively, and enter the desired schedule information in the Schedule parameter. For example, if you want to retrieve a file every day at 1 AM, you would set the Minute field (the first field from the top) to 0, the Hour field (the second field) to *, and the Weekday field (the fifth field) to *, skipping the Day and Month fields. In all likelihood, you would leave the Random minute field (the sixth field) to 0. You would probably also leave the Type field (the seventh field) set to 2, unless you wanted the process and/or service shutdown / re-start operation to be performed immediately when the system starts-up in case the system was shut down when the process and/or service shutdown / re-start operation was scheduled to run (at 1 AM every day). If you left the value of the Fail limit field (the eight field from the top) set to 0, and the Type field set to 2, Scrip 227 would post an event log reporting failure to run because the system was turned off, or the ASI client was not running, the first time such a failure occurs.

12) If you configured Scrip 227 to perform a periodic process / service shutdown / re-start operation, Click on the Submit Changes button. The Changes Recorded page will be displayed.
