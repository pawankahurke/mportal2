Impersonation Support

On NT based systems (MS Windows NT4, 2000, XP, and Server 2003) the ASI client is logged in as local system administrator. This means that it cannot access shared resources on the local network.

In order to enable ASI client's access to shared resources on the local network, we have implemented impersonation. Impersonation lets the ASI client log onto the system where the shared resource is located as a user known by that system enabling it to access that shared resource.

You should enter the known user id and password in Scrip 43, as described below. With this information, the ASI client will be able to access shared resources on the local network as follows:

Impersonation enabled

When the Impersonation enabled parameter is enabled, , using the impersonation user id and password (and domain if necessary) you entered in Impersonation Username, Impersonation Password and Confirmation, and Impersonation Domain parameters in the Scrip 43 configuration page, Scrip 196 can execute patches or software updates that reside on shared resources on other systems on the the network, accessed either as mapped drives(e.g. x:\shared folder), or using UNC (Universal Naming Convention) syntax (e.g. \\server\\shared folder).

How Scrip 196 works

First, Scrip 196 searches the dialog boxes that appear during installation of a patch or software update for buttons, including radio buttons, which makes it work for license agreements.

If no buttons are found, or the Scrip can't read the text from any of the ones that are found, and the Take dialog box default action option is enabled, Scrip 196 sends an Enter command to the dialog box.

If Scrip 196 finds buttons and can read the text that accompanies them, it takes one of the following actions:

  • If something in the Positive response keyword - take default action if found list is found in the text of a button that is not grayed out, and that text doesn't contain anything in the Negative exception keyword - do not take default action if combined with positive response keyword in one phrase list, Scrip 196 clicks on that button,

  • Otherwise, Scrip 196 does not take any action.

The above process is repeated on a five-second cycle. If Max minutes of run time per patch passes, or the maximum number of dialogs is exceeded, Scrip 196 stops the installation process.

Take dialog box default action

If the Take dialog box default action option is enabled, Scrip 196 will always take the default dialog box action overriding any of the other options. While it may work in many cases, this option should be used with care.

For example, If this option is enabled and the default action is Cancel, Scrip 196 will take this action. Obviously, this is not what you would like it to do.

In general, we advise you to leave the Take dialog box default action option disabled, and use the Negative exception keyword - do not take default action if combined with positive response keyword in one phrase and Positive response keyword - take default action if found parameters instead.

Act on dialog boxes of all new processes

If the Act on dialog boxes of all new processes option is enabled, Scrip 196 will take the action corresponding to the value of the Positive response keyword - take default action if found parameter for all dialog boxes.

By default Act on dialog boxes of all new processes is disabled. Scrip 196 will act only on dialog boxes that are created by the patch / software update or subprocesses it creates. If Act on dialog boxes of all new processes is enabled, it will act on dialog boxes created by any process that started after the patch was started. On Microsoft Windows NT based systems (NT4, 2000, XP, and Server 2003), some installations create dialog boxes in the context of the Service Manager and therefore require that the Act on dialog boxes of all new processes option be enabled. If you have a patch or software update where dialog boxes just open and remain on the screen, enabling this option will probably make Scrip 196 act on them.

However, you should be aware that when Act on dialog boxes of all new processes is enabled, you should make sure that the patch or software update runs while the system is not in active use. Otherwise, Scrip 196 will start acting on dialog boxes generated by any programs after the patch or software update starts running.

Names of processes to shutdown

The Names of processes to shutdown parameter can be used to list all the processes you would like Scrip 196 to shut down in an orderly fashion before starting to run patches. The full name of each process to be terminated, including extension, should be entered on a separate line. Full location paths are not required.

This configuration entry should be used with care. For example, you should not use either of the masks * or *.* because they would have the effect of terminating both the ASI Client and other processes (e.g. explorer.exe) that are necessary for executing Scrip 196 (the ASI Client), and keep the system running (explorer.exe).

One or more of the processes running at the time Scrip 196 is scheduled to start that need to be shut down may require user action or response in order to be terminated. For example:

  • Open network connections that in order to be closed require user approval

  • Applications like Microsoft Word that require a user to save or close without saving files that have been changed and not saved at the time Word is terminated, or

  • Information agents like Intuit Quickbooks' qbdagent.exe.

Scrip 196 handles these situations automatically. You can control process shut-down with the Max number of seconds to wait for the process to close and Max number of dialogs to close

The Scrip will also check for processes to shutdown before each patch is installed, in case a process in the Names of processes to shutdown list starts up again.

Max number of seconds to wait for the process to close

This parameter lets you control how long Scrip 196 should wait before shutting down each process that was active at the time the Scrip is scheduled to run.

Max number of dialogs to close

In many instances, requests for user action communicated via dialog boxes are part of process shut down sequences. Scrip 196 handles these requests automatically.

Whenever a dialog box requesting user input appears during a process' shut down sequence, Scrip 196 closes it taking the default action. For example, if the dialog box requests whether a file should be saved or not, and the default response is Yes, Scrip 196 will automatically click on the Yes button.

Patches to apply

The Patches to apply list contains the patch command lines you would like Scrip 196 to run. You can have as many patch command lines defined as you want with each entered on a new line. Patch command lines have the following format, with each parameter in a command line separated by a comma with no spaces:

  • blank, 0 or 1 - if 0 or blank, Scrip 196 does not log dialog boxes produced by patches. If 1, Scrip 196 logs dialog boxes.

  • 9x , nt or blank - the value of this command line parameter tells Scrip 196 which Microsoft Windows operating system family, the patch applies to. It is not case sensitive. If you leave it blank, Scrip 196 will attempt to apply the patch to all systems regardless of operating system. Please note that Microsoft Windows 95, 98, and Me belong to the 9x family while Microsoft Windows NT4, 2000, and XP belong to the NT family.

  • Full path, or URL (see below) of an executable

  • Optionally, command line arguments for the executable. Command line arguments for the executable should be entered as you would enter them in the command line for the executable, each separated by a space. For example: -u -n -Z -Q.

The path(s) of the executables you want Scrip 196 to run must be visible on all the machines you want to run the executable on. This means that if the executable(s) are on a networked drive then the drive must be accessible on all the machines through an appropriately configured drive letter mapping.

For patches/updates residing on networked drives, you can enter either the UNC name of the location, e.g. \\[server name]\[shared resource name]\, or the mapped drive letter if one has been assigned to the networked drive.

Patches/updates can reside on FTP or Web servers (HTTP or HTTPS).

FTP servers can be accessed via a URL scheme that uses a common syntax for IP-based protocols. The syntax of a command line for having Scrip 196 retrieve a patch executable from a FTP server is:

ftp://user:password@host:port/url-path.

The user, password, or port can be omitted, and defaults will be applied. The default user is anonymous. The default password is a ficticious email address (most FTP servers that allow anonymous login either ignore the password or require an email address). The default port is 21, which is the standard FTP port.

Url-path can merely be the name of the remote file (current working directory for the FTP server assumed), or it can denote a particular subdirectory path to the file, e.g., dir/dir/dir/filename.

host is either the name or IP address of the FTP server.

Scrip 196 captures all FTP commands executed when running a patch located on a FTP server. They will be logged automatically onto the ASI server if the patch download process fails.

Web servers can be accessed using HTTP or secure HTTP (HTTPS) via a URL scheme that uses a common syntax for IP-based protocols. The syntax of a command line for having Scrip 196 retrieve a patch executable from a Web server is:

http://user:password@host/url-path, or https://user:password@host/url-path

For publicly accessible Web pages, the user and password are not necessary, and should be omitted.

Url-path can merely be the name of the remote file (current working directory for the Web server assumed), or it can denote a particular subdirectory path to the file, e.g., dir/dir/dir/filename.

host is either the name or IP address of the Web server.

Scrip 196 captures all HTTP commands executed when running a patch located on a Web server. They will be logged automatically onto the ASI server if the patch download process fails.

Please note that capitalization of the name of the patch/software update executable in the path parameter of a Scrip 196 command line needs to match capitalization of the patch/software update name on the FTP or HTTP/HTTPS server. Otherwise, Scrip 196 will report a file not found error. Generally speaking, to minimize the probability of an error, it is advisable to use all-lower case names.

Each ASI server includes a folder called patches located in the publicly accessible https://ASIserverURL/main/pub directory. You can use the patches folder as a convenient location from which patches and software update executables can be downloaded by Scrip 196 using HTTP/HTTPS, without requiring authentication. A sample Scrip 196 command line for installing a patch located in the folder patches would look something like this:

1,,https//ASIserverURL/main/pub/patches/patch.exe

On each ASI server we have also installed an FTP server and created a a user id and password with upload access to the patches folder to upload patches and installation executables to this folder.

Please note that you should not use batch files in the Patches to apply list. Scrip 196 operates on the dialog boxes associated with the executable listed in the Patches to apply list. If it were a batch file, then it would wait to close the MS-DOS window opened by the batch file, not act on the dialog boxes generated by the executables contained in the batch file.

If you make a mistake in entering the path for the executable you want Scrip 196 to install, the Scrip will report the error in an event log uploaded to the ASI server.

Please keep in mind that the following characters cannot be used in an FTP or HTTP URL:

< > # % { } | ~ ^ [ ] ' \

Stop on an error

If something goes wrong during the execution of Scrip 196, e.g. one of the executables in the list is not found, you can choose to continue with the execution of the other executables (this parameter would be disabled), or stop execution of the Scrip (this parameter would be enabled).

Max minutes of run time per patch

With the Max minutes of run time per patch parameter, you can control execution of an executable preventing it from running away and hanging the system. If the executable runs longer than the value in minutes you give to this parameter, the Automated Support Infrastructure (ASI) client will terminate its execution. A value of 0 means that the executable can run without a time limit.

Please note that the time spent downloading a patch from an FTP or Web server, is not included in the Max minutes of run time per patch parameter unless the software update or patch executable performs a donwload operation as part of the patch/software update process.

Patches installed

With the Patches installed list, Scrip 196 keeps track of patches it previously installed. As a result you don't have to remove patches/software updates from the Patches to apply list because the Scrip will not try to install a patch/softwatre update that is already installed. If you need to force an update, all you need to do is remove the entry for the patch/software update from the Patches installed list. Please note that the entire line needs to be removed not just the patch/software update name.

Apply patches now

With the Execute button, you can run Scrip 196 on demand, even if it's not enabled. In Scrip 196, the Execute button can be used to execute the Scrip on the local system or on all supported systems.

Schedule

The Schedule parameter has a command line with eight fields, each entered on a new line.

The table below describes each field in detail.

Field Number

Field Name

Field Values and Description

1

Minutes

0-59; a value of * means that the Scrip should run every minute

2

Hours

0-23; a value of * means that the Scrip should run every hour

3

Day of the Month

1-31, depending on the month; a value of * means that the Scrip should be run every day.

4

Month

1-12; a value of * means that the Scrip should be run every month.

5

Day of the Week

0, Sunday - 6, Saturday; a value of * means that the Scrip should be run every day of the week.

6

Minute Randomizer

N; the size of the time interval in minutes over which the Scrip should be randomly executed. We multiply N*60, then calculate a random number X between 1 and N*60, and run the Scrip X seconds after its scheduled time.

7

Conditional Execution

N; N=1 means that if the Scrip is past due, it will be run immediately, N=2 means that If the Scrip is past due, it will be run at the next scheduled execution time. If the value is set to 2, the actual Scrip execution time will be within 3 minutes (either way) of the scheduled time. This is necessary to prevent the case where a process will not be executed if it is a little slow to start, or if the scheduler starts it a few seconds too early.

8

Failed Run Limit

N, where N is the number of times that Scrip execution is allowed to fail before notification is sent to the support provider; N=0, the Scrip will NEVER report a failure, N=1, the Scrip will report EVERY failure, N=2, the Scrip will report if it failed twice in a row, and so on.



The Minutes, Hours, Day of the Month, Month, and Day of the Week fields can have multiple values and ranges of values separating them with commas.

For example, a 1,3,5 entry for the Minutes parameter would mean that a Scrip would be executed at minutes 1, 3, and 5.

A 10-14 entry for the Hours parameter would mean that a Scrip would be executed starting at 10 AM and ending at 2 PM.

Multiple value and range entries can be used together separated by commas.
