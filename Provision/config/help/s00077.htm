Scrip 77 Configuration

Configuration Parameters and their default values

  • Scrip enabled

  • Report error logs enabled

  • Report warning logs disabled

  • Report informational logs disabled

  • Report successful access security audits disabled

  • Report failed access security audits disabled

  • Report unknown logs disabled

  • Polling interval (minutes, 0=none) 0

  • Event logs to monitor for changes Application, System, Security

Report unknown logs

When the parameter Report unknown logs is enabled, Scrip 77 will report the addition of undocumented logs to the event log, and their content.

Polling interval (minutes, 0=none)

The Polling interval (minutes, 0=none) configuration parameter can be used to work around the problem described in the following Microsoft knowledge base article:
http://support.microsoft.com/default.aspx?scid=kb;en-us;Q245609

Based on the problem described in the article referred to above, the worst that could happen is that there would be a delay in seeing an event log until the next time that either another one happened, or the system re-started. The way Scrip 77 works, you will NEVER miss seeing an event log.

However, we felt that the best approach would be to give you the option to have Scrip 77 based on event detection (the default and preferred approach, with the Polling interval (minutes, 0=none) configuration parameter set to 0), or based on polling.

With polling, if you have some events that are really time critical, and it is imperative that you see every event at most 10 minutes after it happens, for example, then you will be able to do that. All you need to do is set the Polling interval (minutes, 0=none) configuration parameter to 10.

If you can live with the occasional problem in the operating system that limits our ability to deliver real-time information, then you can leave the Polling interval (minutes, 0=none) configuration parameter set to 0, which is the way we suggest running it.

Event logs to monitor for changes

In addition to the default entries, you can add the name of custom event logs to the Event logs to monitor for changes list, each on a separate line.

Security audits

Please note:

Success audit - Success audit events are security events that occur when an audited access attempt is successful. For example, a successful logon attempt is a success audit event.

Failure audit - Failure audit events are security events that occur when an audited access attempt fails. For example, a failed attempt to open a file is a failure audit event.
