How Scrip 232 works

Scrip 232 is enabled by default. Whenever an attempt is made to change the configuration settings it, or the objects it monitors, Scrip 232 notifies the support provider by logging the event onto the ASI server.

The first time Scrip 232 is enabled or run it collects all the configuration settings and creates backup copies of all the system objects and screen savers it monitors. This collection of information represents the baseline against which it will compare any changes to monitored system configuration settings and system objects it detects from then on.

Because this Scrip protects a system from unwanted / malicious changes both to configuration settings and certain special objects, which are inherently very different from each other, the way they are protected is also different.

System objects

For objects, Scrip 232 creates a hidden directory in the ASI client directory called 232. When the Scrip first runs it stores in this hidden directory backup files of autoexec.bat, explorer.exe, hosts, and userinit.exe. These back-up files have the name of the original file with the .bak extension.

Please note that on systems running Microsoft Windows NT based operating systems (NT4, 2000, XP, or Server 2003), there may or may not be an autoexec.bat file. If there isn't one, and someone creates an autoexec.bat if scrip 232 is configured to reject changes (either silently, by dialog box time-out, or by user action) then it simply rejects the new autoexec.bat in order to maintain the system in the same state. If autoexec.bat is created and Scrip 232 is configured to accept changes (either silently, by dialog box time-out, or by user action), Scrip 232 will generated a backup copy of the new autoexec.bat file, and store it in directory 232.

On systems running a version of Microsoft Windows 95, 98, or Me operating systems Scrip 232 does not generate copies of userinit.exe and explorer.exe because on these systems the security hole that would allow an intruder to replace these critical files (that run on user logon) exists only in the registry on NT based systems.

Scrip 232 monitors autoexec.bat, explorer.exe, hosts, and userinit.exe for tampering using checksums. Should one of these files change, Scrip 232 will use the corresponding backup file in directory 232 to restore it, if Scrip 232 is configured to reject such changes (either silently, by dialog box time-out, or by user action). If a file changes and the Scrip is configured to accept changes, then the backup file is updated.

If Scrip 232 is configured to disable changes to autoexec.bat, explorer.exe, hosts, and userinit.exe, in addition to restoring the file, using its back-up copy stored in directory 232, it archives the new, disabled, version in directory 232 as well. Archived copies of disabled files have the name of the original file with the .dbl extension. This will male it possible to examine the changed file at your convenience.

System configuration settings

The other items scrip 232 monitors are configuration settings. The current value of all these settings are saved in the ASI client's persistent state repository so that if one of these settings were to change, and the Scrip is configured to reject configuration changes, the previous value of the configuration setting can be restored.

Screen savers

Scrip 232 handles screen savers both like system configuration settings, and like system objects. This is because there are two ways unwanted / malicious code can be introduced using screen savers:

  • A rogue screen saver containing a virus can be copied to a system and then the currently selected screen saver can be programmatically changed to the viral one. To prevent this Scrip 232 stores the name of the currently selected screen saver in the ASI client's persistent state repository and monitors it to see if someone or something changes it. If a change is detected Scrip 232 will execute the action you configured it to take, or the one selected by the user, if the user interface is enabled.

  • A rogue application can also get the name of the current screen saver. It can then rename an intrusion executable to that screen saver name and then wait for a time when the screen saver is not active and overwrite it with the infected copy. The next time the screen saver is activated, the intrusion executable will also be activated. To prevent this type of intrusion Scrip 232 keeps a backup copy of the currently selected screen saver file as it does for autoexec.bat, explorer.exe, hosts, and userinit.exe. The Scrip also keeps a copy of the currently selected screen saver's checksum in the ASI client's persistent state repository. It monitors the screen saver file. If it changes, it takes another checksum and compares it with the checksum it stored in persistent state. If the checksums do not match then it's clear that some tampering has occurred. Scrip 232 will process the change. If you configured it to reject changes, the backup copy of the selected screen saver will overwrite the changed one. If, instead, the Scrip is configured to accept changes, the Scrip's copy of the checksum is updated, and the backup copy of the screen saver file is updated. If the user interface is enabled, the user will be able to take the same actions (accept, disable, reject).

Scrip 232 only keeps a backup copy of the currently selected screen saver but it keeps the checksums of all the screen savers on a system. If an inactive screen saver's file gets replaced a log is posted on the ASI server signaling the event but no dialog box will appear (if the user interface is enabled), and no other action will be taken.

Making changes to areas protected by Scrip 232

When you want to make configuration changes on one or more systems where Scrip 232's disable or reject configuration options are enabled, you should follow the following steps:

  • Disable Scrip 232 on one, a subset, or all systems

  • Make the desired configuration change

  • Re-enable Scrip 232 on the system(s) you had disabled it on. As soon as you do that, Scrip 232 runs and rebuilds the list of the programs configured to run at start-up.

Attempt to terminate processes that try to change protected registry items

When the Attempt to terminate processes that try to change protected registry items configuration parameter is enabled, if Scrip 232 is able to identify the process that attempted to make a registry change (and the registry change was rejected or disabled), it will attempt to stop that process if it is included in the List of processes that may be terminated configuration parameter, and is not in the List of processes that will never be terminated configuration parameter.

Attempt to delete processes that try to change protected registry items (after termination)

When the Attempt to delete processes that try to change protected registry items (after termination) configuration parameter is enabled, if Scrip 232 successfully terminates the process, or the process is no longer running (but it would have been terminated if it were) the executable file of the process will be deleted. If the file could not be deleted, Scrip 232 configures the system to delete it upon the next system re-start.

Attempt to quarantine processes that try to change protected registry items (after termination)

When the Attempt to quarantine processes that try to change protected registry items (after termination) configuration parameter is enabled, if Scrip 232 successfully terminates the process, or the process is no longer running (but it would have been terminated if it were) the executable file of the process is moved to a hidden folder called quar located in the ASI client directory. If the original file could not be removed, it is marked for deletion upon re-start (the copy operation to the quarantine almost always succeeds).

List of processes that may be terminated

The List of processes that may be terminated configuration parameter contains a list of full executable paths or individual executable names with optional '*' wildcards defining the processes Scrip 232 is permitted to terminate, quarantine, or delete. Each item in the configuration parameter is entered on a new line.

List of processes that will never be terminated (If no path is specified, the program is allowed to run from any directory)

The List of processes that will never be terminated configuration parameter contains a list of full executable paths or individual executable names with optional '*' wildcards defining the processes Scrip 232 will never take a terminate, quarantine, or delete action on. Each item in this configuration parameter is entered on a new line.

The List of processes that will never be terminated configuration parameter supersedes the List of processes that may be terminated configuration parameter. This means that if you enable one of the configuration parameters that will cause Scrip 232 to terminate, delete, or quarantine a process, the Scrip will first attempt to find a match with a process in the list of processes you entered in the List of processes that will never be terminated configuration parameter. If it finds a match, the Scrip will ignore the contents of the List of processes that may be terminated configuration parameter. The List of processes that will never be terminated configuration parameter takes priority, so a process that is included in both the List of processes that may be terminated, and the List of processes that will never be terminated configuration parameters will never be terminated, quarantined, or deleted.

Please note that if you do not specify the full path for a process you want Scrip 232 to terminate, the Scrip will assume that the process can reside anywhere on the system.

How Scrip 232 matches processes attempting changes with processes in processes-to-be terminated and never-to-be terminated configuration parameters

Suppose the List of processes that may be terminated configuration parameter has the * default entry, meaning that all processes that attempt to make a change to a system's configuration areas protected by Scrip 232 will be terminated, if the Attempt to terminate processes that try to change protected registry items configuration parameter is enabled.

If the process attempting to change a registry key in an area protected by Scrip 232 is D:\regedit.exe, here is how different entries in the List of processes that will never be terminated configuration parameter would compare resulting in a match preventing D:\regedit.exe from being terminated, or not in which case D:\regedit.exe would be terminated:

  • D:\regedit.exe – Matches, D:\regedit.exe would not be terminated.

  • regedit.exe – Does not match, D:\regedit.exe would be terminated.

  • *regedit.exe – Matches, D:\regedit.exe would not be terminated.

  • *\regedit.exe – Matches, D:\regedit.exe would not be terminated.

Since the full path name (D:\regedit.exe) is used for the comparison, the second entry would not match. We would recommend using the fourth entry to prevent processes such as D:\virusregedit.exe from matching an entry in the List of processes that will never be terminated configuration parameter if you want to use process names as entries in the List of processes that will never be terminated configuration parameter.

Permitted items function

How it works

When it detects an attempt to add, delete, or change an item to a system's start-up environment, Scrip 232 compares the start-up item it detects with the content of the Permitted items configuration parameter. If it finds a match, it will allow the addition, deletion, or change of the start-up item. Otherwise, it will disable or reject the addition, deletion, or change, depending on how it is configured.

Scrip 232 processes the list of items in the Permitted items configuration parameter sequentially. It tests an item for one of the following potential matches:

  1. Checks if the start-up item matches exactly an item in the Permitted items configuration parameter.

  2. The start-up item is a registry entry and its registry branch matches a RegBranch (registry branch) entry in the Permitted items configuration parameter.

  3. The start-up item is a registry entry and its key variable name, and/or key variable value match a RegValString (registry key variable value sub-string) entry

  4. A portion of the start-up item matches a Keyword entry in the Permitted items configuration parameter

Permitted items

You can add entries to the Permitted items configuration parameter manually, and by using the Populate “Permitted items” list now action. Permitted items configuration parameter entries can be of the following type:

Registry key

RegKey,<user name>,<key name>,<key variable name>,<key variable value>

Registry key variable value sub-string

RegValString,<user name>,<key name>,<key variable name>,<key variable value>

Registry branch

RegBranch,<any portion of a registry key path except for registry key <key variable name> and <key variable value>>

Extension object

ExtObj,<User name>,<friendly name>,<key name>,<key variable name>,<key variable data>,<associated DLL>

Keyword

Keyword,<keyword value>

Ini file

IniFile,<User name>,<.ini file (either win.ini or system.ini) path>,<.ini file section>,<.ini file variable name>,<.ini file variable value>

Text file

TxtFile,<User name>,<Text file path>,<Text file line>



Please note that in any of the above entries, there are no spaces before or after commas.

Populate "Permitted items" list now

If you click on the Populate “Permitted items” list now Execute button, Scrip 232 will populate the Permitted items configuration parameter with the content of the Currently enabled startup items on this machine – known Scrip 240 configuration parameter.

Please note that taking the Populate “Permitted items” list now action will populate the Permitted items configuration parameter with items that will be used by the permitted items function for exact matches. In other words, Scrip 232 will allow deletion, or changes only of items matching exactly entries in the Currently enabled startup items on this machine – known Scrip 240 configuration parameter.

Taking the Populate “Permitted items” list now action will not overwrite the content of the Permitted items configuration parameter.

Since the Permitted Items configuration parameter can be assigned to one system, a group of systems, or an entire site, when populate the Permitted Items configuration parameter using the Populate Permitted items Execute button, its content can be propagated automatically to all systems at a site, or a group of systems.

Standard interface

User interface enabled

The User interface enabled configuration option lets you enable or disable the user interface (end-user dialog box). When the user interface is enabled, and a potential configuration change is detected, the change is reported in a dialog box and the user is given the option to accept, disable, or reject the change. The user's choice overrides the Do not allow system configuration changes, Do not allow the software update operations, Disable rather than reject system configuration changes, and Disable rather than reject software update operations configuration options, even if they are enabled. By default, this option is disabled.

If the end-user does not take any action when the dialog box appears, after 60 seconds two events can occur:

  • If some or all of the Do not allow system configuration changes, Do not allow the software update operations, Disable rather than reject system configuration changes, and Disable rather than reject software update operations configuration options are enabled, Scrip 232 will take action accordingly, with the Disable configuration options overriding the Do not allow options, if both are enabled.

  • If none of the Disable or Do not allow options are enabled, Scrip 232 will take no action.

Do not allow system configuration changes

When the Do not allow system configuration changes configuration option is enabled, Scrip 232 will reject potential configuration changes it detects. By default, this option is disabled.

Do not allow software update operations

When the Do not allow software update operations configuration option is enabled, Scrip 232 rejects the copies of autoexec.bat, explorer.exe, and userinit.exe which an intruder tries to replace the original with. By default, this option is disabled.

Please note that when installing a Microsoft Windows software update or patch, the Do not allow software update operations configuration option must be disabled until after system restart. If it is not disabled, the system may end up in a unknown, unstable state.

Disable rather than reject system configuration changes

When the Disable rather than reject system configuration changes configuration option is enabled, Scrip 232 places potential configuration changes it detects in the 232 directory. If this configuration option is enabled it overrides the Do not allow system configuration changes configuration option, even if it is enabled. By default, this option is disabled.

The Disable rather than reject system configuration changes configuration option has no effect unless the Do not allow system configuration changes configuration option is enabled.

If both the Disable rather than reject system configuration changes and Do not allow system configuration changes options are enabled and the User interface enabled option is not, the Disable rather than reject system configuration changes option overrides the Do not allow system configuration changes option.

Disable rather than reject software update operations

When the Disable rather than reject software update operations configuration option is enabled, Scrip 232 place copies of autoexec.bat, explorer.exe, and userinit.exe which an intruder tries to replace the original files with in the 232 directory. If this parameter is enabled it overrides the Do not allow software update operations parameter, even if it is enabled. By default, this option is disabled.

The Disable rather than reject software update operations configuration option has no effect unless the Do not allow software update operations configuration option is enabled.

If both the Disable rather than reject software update operations and Do not allow software update operations options are enabled and the User interface enabled option is not, the Disable rather than reject software update operations option overrides the Do not allow software update operations option.

Please note that when installing a Microsoft Windows software update or patch, the Disable rather than reject software update operations configuration option must be disabled until after system restart. If it is not disabled, the system may end up in a unknown, unstable state.

Note on configuration changes

When scrip 232 runs for the first time it checks the value of several registry keys, including the shell open commands, to see if they are set to the default values that were set when the operating system was installed. Typically, these registry key values should not be changed. If they are, it can be the result of a possible virus attack. However, there are cases where these registry key values are changed by the installation of a legitimate application.

Scrip 232 detects but can't process these changes because when the ASI client first runs even though the Scrip is enabled by default, the Disable configuration change and Reject configuration change configuration options, as well as the User interface enabled option are disabled. This means that when it first runs, if Scrip 228 detects that the value of one of these critical registry keys is not set to its default, it just reports this fact so that you have the option to check and see if the installation of a legitimate application caused the change(s).

Advanced interface

The advanced interface lets you select the action the Scrip should take depending on the area being attacked. The possible actions are:

  • 1 - Scrip 232 will accept changes, taking no action. Changes will still be logged.

  • 2 - Scrip 232 will reject changes, restoring original conditions.

  • 3 - Scrip 232 will reject changes, and if possible, will allow for rejected changes to be re-enabled at a later time.

  • 4 - Scrip 232 will display user interface and take action 1 if dialog box times out.

  • 5 - Scrip 232 will display user interface and take action 2 if dialog box times out.

  • 6 - Scrip 232 will display user interface and take action 3 if dialog box times out.
