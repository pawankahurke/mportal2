Scrip 225 Configuration

Configuration parameters – default values

  • Scrip disabled

  • Continuously monitor for changes enabled

  • Monitored items empty

  • When enabled, by default Scrip 225 runs daily at 2.25 AM, once per hour between 6.25 AM and 6.25 PM, and at 9.25 PM

  • Check monitored items now

Continuously monitor for changes

When the Continuously monitor for changes configuration parameter is enabled, Scrip 225 will detect changes to the directories and files listed in the Monitored items configuration parameter as they happen and will prevent changes to files being monitored, if so configured. When this configuration parameter is disabled, Scrip 225 will only check for changes when it is scheduled to run. Please note that enabling this configuration parameter can impact performance if the directories being monitored contain many items, and are very active.

Monitored items

The Monitored items configuration parameter contains the list of directories and files monitored by Scrip 225. Each item to be monitored is entered on a new line. The format of an item in the list is:

  • 0 (disable) or 1 (enable) - this makes it possible to temporarily stop monitoring selected items without deleting the corresponding monitored item command line from the list.

  • Sub-directory monitor flag - If it is set to 1, Scrip 225 will monitor all sub-directories. When this flag is set to 0, Scrip 225 will ignore any sub-directory contained in the directory being monitored.

  • Protection - If this flag is set to 1 Scrip 225 will prevent any changes to be made to files being monitored. Please note that this function applies only to files. When this flag is set to 0, Scrip 225 will only detect and report changes made to monitored items.

  • Directory name - The full path for a directory you want Scrip 225 to monitor, or a directory specification that may have an environment variable in it, or wild cards (see section on directory wild cards at the end of this document). . For example:

    • c:\windows\system32

    • %windir%

    • %windir%\prefetch

  • File mask - The file mask to use for identifying items to monitor, or single file name to be monitored by Scrip 225.

When specifying file masks, you can use the standard Microsoft MS-DOS wild card characters ? and * in both the file name and extension.

Some examples of Monitored items parameter list items are:

  • 0,0,0,c:\,*.txt - Scrip 225 will ignore this entry because it is disabled

  • 1,0,1,%windir%\system32,*.dll - This command line is enabled. It covers the system32 sub-directory in the Windows directory and monitors and prevent any changes to be made to existing dll files

  • 1,1,0,C:\Program Files\Microsoft Office\Templates\,*.dot - This command line is enabled. It covers Templates directory and all its sub directories. It monitors files of type dot (Microsoft Word template files) but does not prevent changes from being made to them.

Please note that when a command line in the Monitored items configuration parameter is disabled, the stored information about the directory and its contents is removed. When a command line is enabled, the information about the directory and its contents is re-generated.

Note about Scrip 225 execution

Scrip 225 runs when one of the following events occurs:

  • When the ASI client starts up. When this happens it checks for changes

  • When you click on the execute button to the right of the Check monitored items now label. When this happens it checks for changes.

  • At its scheduled time. When this happens it checks for changes.

  • When monitored item(s) change. When this happens it checks for changes.

  • When one of its configuration parameter changes. When this happens it updates the configuration.

  • When it is enabled/disabled. When this happens it updates the configuration.

Checking for changes means:

  • Determining if any files or directories have been removed, added or altered, and

  • Updating the stored information about any item that has changed

Updating the configuration means updating the stored information about monitored item(s) to match their current state. If any existing stored information for monitored item(s) is still relevant, then it is re-used.

If you enabled protection for a command line in the Monitored items parameter(value set to 1), instead of just updating the stored information (checksum contained in the ASI client's persistent state repository) for the item(s) in that command line, when an item changes, Scrip 225 will restore the item to its pre-change state by overwriting the changed item with the copy that was stored in the hidden directory 225 contained in the ASI client directory when the item was first added to a (part of) Monitored items parameter command line.

Please note that Scrip 225 will restore any protected file or directory with the backup copy stored in the not only if an attempt is made to change them but also if they are deleted.

If you change the copies of the files and/or directories you have configured Scrip 225 to monitor (and protect) that are stored in directory 225, the Scrip will not update the changed item's (or items') checksum information until it is disabled and then re-enabled, or the ASI client is restarted.

Check monitored items now

With the Execute button, you can run Scrip 225 on demand, even if it's not enabled. In Scrip 225, the Execute button can be used to execute the Scrip on the local system or on all supported systems.

Schedule

The Schedule parameter has a command line with eight fields, each entered on a new line.

The table below describes each field in detail.

Field Number

Field Name

Field Values and Description

1

Minutes

0-59; a value of * means that the Scrip should run every minute

2

Hours

0-23; a value of * means that the Scrip should run every hour

3

Day of the Month

1-31, depending on the month; a value of * means that the Scrip should be run every day.

4

Month

1-12; a value of * means that the Scrip should be run every month.

5

Day of the Week

0, Sunday - 6, Saturday; a value of * means that the Scrip should be run every day of the week.

6

Minute Randomizer

N; the size of the time interval in minutes over which the Scrip should be randomly executed. We multiply N*60, then calculate a random number X between 1 and N*60, and run the Scrip X seconds after its scheduled time.

7

Conditional Execution

N; N=1 means that if the Scrip is past due, it will be run immediately, N=2 means that If the Scrip is past due, it will be run at the next scheduled execution time. If the value is set to 2, the actual Scrip execution time will be within 3 minutes (either way) of the scheduled time. This is necessary to prevent the case where a process will not be executed if it is a little slow to start, or if the scheduler starts it a few seconds too early.

8

Failed Run Limit

N, where N is the number of times that Scrip execution is allowed to fail before notification is sent to the support provider; N=0, the Scrip will NEVER report a failure, N=1, the Scrip will report EVERY failure, N=2, the Scrip will report if it failed twice in a row, and so on.



The Minutes, Hours, Day of the Month, Month, and Day of the Week fields can have multiple values and ranges of values separating them with commas.

For example, a 1,3,5 entry for the Minutes parameter would mean that a Scrip would be executed at minutes 1, 3, and 5.

A 10-14 entry for the Hours parameter would mean that a Scrip would be executed starting at 10 AM and ending at 2 PM.

Multiple value and range entries can be used together separated by commas.

Directory path wild card support

Certain repetitive system maintenance and management tasks require access to sub-directories that have the same name but have different parent directories. For example, clean-up of temp directories on systems running the Microsoft Windows NT based operating systems (NT4, 2000, XP, and Server 2003), requires access to each user's temp directory located in [drive}:\Documents and Settings\<user name>\Local Settings\Temp.

In order to make it easier for you configure Scrips to perform such repetitive tasks, we have implemented wild card directory support, allowing wild cards anywhere in a directory path. This means that in the example above, specification of the path for the temp directory on all fixed and network drives would be reduced to a single line:

*:\Documents and Settings\*\Local Settings\Temp\

Wild card directory support applies to any Scrip where you may need to specify a directory path, including Scrips 60 (and 217-221), 61, 100, 196, 222, 225, 229, and 230. It also extends to devices, meaning that you can use the wild card character for devices including:

  • Fixed drives (like hard disk partitions)

  • Removable drives (like floppies)

  • Network drives

  • CD-ROMs

  • RAM disk drives

  • Unknown drive types

For example:

\\*\c:\Windows\**\temp

would scan the temp directory found zero or more levels below the Windows directory on the c: drive on all reachable systems on the local network.

The notation used for wild card directory support is slightly different from the MS-DOS wild card notation. It consists of either * or **:

  • * matches exactly one directory level.

  • ** matches zero or more directory levels.

For example:

\Documents and Settings\**\Local Settings\Temp\

would cover all directory levels where the end path consists of \Local Settings\Temp\ including:

\Documents and Settings\Local Settings\Temp\

and

\Documents and Settings\<user name>\Local Settings\Temp\

while

\Documents and Settings\*\Local Settings\Temp\

would cover only one directory level above \Local Settings\Temp\

You can use multiple instances of * and/or ** in directory path specifications. When the directory wild card support software finds instances of multiple wild cards, it performs left-to-right scan that looks through directories

recursively. Please note that his means that multiple ** or * in a directory path specification can cause a lot of multiple scans through the same directory structure, which will take a lot of time, but it will return the correct result.
