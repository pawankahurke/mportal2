Reporting Thresholds

When configuring Scrip 96, users can set reporting thresholds for processor usage, Interrupts per second, and Queue length. They can be used to alert the support provider about a system with an under-powered processor, given the application mix run by the end-user, or a potentially problematic application. Threshold settings also have the effect to regulate the Scrip's logging activity.

The reporting threshold for Interrupts per second can be any number. When set to 0, Scrip 96 will log processor statistics every fifteen, the default logging period. When it is set to a very large number, typically more than , Scrip 96 will log processor utilization statistics very rarely.

The reporting threshold for Queue length can be any number. When set to 0, Scrip 96 will log processor utilization statistics every fifteen minutes, the default logging period. When it is set to a very large number, that varies depending on the processor and operating system combination, Scrip 96 will log processor utilization statistics very rarely.

Reporting threshold for processor usage can vary between 100 and 0. When a reporting threshold is set to 0, in practice this means that Scrip 96 will log processor utilization statistics every 15 minutes, the defaul logging period. When the reporting threshold is set to X, where X is between 0 and 100, this means that for that Scrip 96 will log processor utilization statistics every 15 minutes, if Usage percentage exceeds X. When processor usage reporting threshold is set to 100, in practice this means that for that Scrip 96 will log processor utilization statistics very rarely.

Of the three metrics, experience shows that a high Queue length value typically represents the most reliable indicator of a potential processing capacity shortage as it tends to indicate that a large number of processes are waiting for processor capacity in order to execute. This, in turn, leads to degraded application performance.

You should note that Scrip 96 will log the value of all processor metrics whenever the threshold value of any one metric is exceeded.

Limit number of processes logged to

When the Report running processes configuration parameter is enabled, Scrip 96 will report detailed information about a number of processes equal to the value of the Limit number of processes logged to configuration parameter in descending order by reported processor utilization. The default value of the Limit number of processes logged to configuration parameter is 10. If the value of the Limit number of processes logged to configuration parameter is set to 0, and the Report running processes configuration parameter is enabled, Scrip 96 will report detailed information about all processes running on the system at the time Scrip 6 posted its event log.

Report running processes

If you enable the Report running processes configuration parameter, when Scrip 96 runs the event log it posts on the ASI server will include the list of processes running at the time its execution.

Please be aware that event logs including the list of running processes can be fairly large (>50 Kbytes). If Scrip 96 thresholds are set too low, the number of Scrip 96 logs generated can number in the thousands per system per day. If the Report running processes configuration parameter is enabled, the result can be high network connection and mass storage capacity utilization on the ASI server.

Report CPU usage now

With the Execute button, you can run Scrip 96 on demand, even if it's not enabled. In Scrip 96, the Execute button can be used to execute the Scrip on the local system or on all supported systems.

Schedule

The Schedule parameter has a command line with eight fields, each entered on a new line.

The table below describes each field in detail.

Field Number

Field Name

Field Values and Description

1

Minutes

0-59; a value of * means that the Scrip should run every minute

2

Hours

0-23; a value of * means that the Scrip should run every hour

3

Day of the Month

1-31, depending on the month; a value of * means that the Scrip should be run every day.

4

Month

1-12; a value of * means that the Scrip should be run every month.

5

Day of the Week

0, Sunday - 6, Saturday; a value of * means that the Scrip should be run every day of the week.

6

Minute Randomizer

N; the size of the time interval in minutes over which the Scrip should be randomly executed. We multiply N*60, then calculate a random number X between 1 and N*60, and run the Scrip X seconds after its scheduled time.

7

Conditional Execution

N; N=1 means that if the Scrip is past due, it will be run immediately, N=2 means that If the Scrip is past due, it will be run at the next scheduled execution time. If the value is set to 2, the actual Scrip execution time will be within 3 minutes (either way) of the scheduled time. This is necessary to prevent the case where a process will not be executed if it is a little slow to start, or if the scheduler starts it a few seconds too early.

8

Failed Run Limit

N, where N is the number of times that Scrip execution is allowed to fail before notification is sent to the support provider; N=0, the Scrip will NEVER report a failure, N=1, the Scrip will report EVERY failure, N=2, the Scrip will report if it failed twice in a row, and so on.



The Minutes, Hours, Day of the Month, Month, and Day of the Week fields can have multiple values and ranges of values separating them with commas.

For example, a 1,3,5 entry for the Minutes parameter would mean that a Scrip would be executed at minutes 1, 3, and 5.

A 10-14 entry for the Hours parameter would mean that a Scrip would be executed starting at 10 AM and ending at 2 PM.

Multiple value and range entries can be used together separated by commas.
