How Scrip 27 works

Scrip 27 is enabled by default. In its default configuration, it only monitors and reports changes to a system's startup areas, and addition of unwanted pending file rename operations. By default, it does not prevent any of these changes from taking place.

When it is configured to prevent changes to a system’s start-up environment, or pending file rename operations, whenever an attempt is made to change a system's start-up environment, add a pending file rename operation, or remove an item from a system’s start-up environment, Scrip 27 posts an event log onto the ASI server, and prevents the change from taking place.

Start-up executables

When Scrip 27 is enabled or run for the first time, it builds a list of all the programs that run at system start-up by extracting the relevant information from the registry keys and file folders that are used to control which programs are run when the system starts. These include the Startup folder, the win.ini file, the system.ini file, and the registry keys described in

http://support.microsoft.com/default.aspx?scid=KB;en-us;q137367

, and

http://support.microsoft.com/default.aspx?scid=kb;EN-US;314866

When the Scrip detects an attempt to add a new executable to run at system start-up, it either deletes the executable, disables it or ignores it (the Cancel action in the dialog box when the user interface is enabled).

Depending on the configuration options you select, either the user is prompted to take one of these actions, if the user interface is enabled, or the Scrip automatically performs the desired action.

Cancel allows the addition of the executable to the system's start-up environment. Delete prevents the addition of the executable to the system's start-up environment by deleting its entry in the registry key or in the Startup folder, the win.ini file, or the system.ini file. Disable also prevents the addition of an executable to a system's start-up environment but stores a copy of the executable's entry in a disabled key in the registry, in the [disabled] section in, win.ini, and system.ini, or in the disabled start-up folder so that you can look at the rejected item at your convenience. The Disable action also gives you the opportunity to re-enable the pending file rename operation by moving it to the disabled registry key or section back to the real registry key or section.

Note that the Scrip does not monitor the RunOnce registry keys, since those are primarily used for system operations, and will not cause changes that permanently affect system performance.

Due to a limitation in Windows 95, this Scrip cannot be triggered by registry key changes on that operating system. Because of that limitation, and in order to avoid missing a change when the client is not running, on systems running Microsoft Windows 95 the Scrip also runs at system start-up, and whenever the ASI client starts.

Pending file rename operations

Scrip 27 also monitors, and lets you disable or remove pending file rename operations, which execute at system start-up. Operating system files that are loaded at system start-up can't be deleted, replaced, or renamed when the operating system is running because it locks them. Pending file rename operations provide a mechanism to delete, replace, or rename these files by executing the action at system start-up before the files get loaded by the operating system. This mechanism can be exploited by someone with malicious intent.

The first time it is enabled or run, Scrip 27 builds a list of the pending file rename operations that are scheduled to run at system startup. These operations are stored in the PendingFileRenameOperations registry key on Microsoft Windows NT based systems (NT4, 2000, XP, and Server 2003) and in the [rename] section in wininit.ini on Microsoft Windows 9x based systems (95, 98, 98SE, and Me).

Scrip 27 updates its list of pending file rename operations whenever such an operation is added to the registry or to wininit.ini. It also empties the list whenever the system starts up. Pending file rename operations work in a similar way to the RunOnce registry keys. A pending file rename operation is executed when the system is (re)started. Scrip 27 removes the operation from the registry or wininit.ini before it can be executed.

When the Scrip detects that one of these operations is scheduled, it either deletes the pending file rename operation, disables it or ignores it (the Cancel action in the dialog box when the user interface is enabled). Depending on the configuration options you select, either the user is prompted to take one of these actions, if the user interface is enabled, or the Scrip automatically performs the desired action.

Cancel allows the pending file rename operation to run as scheduled when the machine next reboots. Delete prevents the operation from running by deleting the registry key or entry in wininit.ini. Disable also prevents the operation from running but stores a copy of the operation in a disabled key in the registry or in a disabled section in wininit.ini so that you can look at the rejected operation at your convenience. The Disable action also gives you the opportunity to re-enable the pending file rename operation by moving it to the disabled key or section back to the real key or section.

System update operations

As described above, the pending file rename operation mechanism is designed to replace or delete files in use and locked by the operating system when a system is running. It can be used to delete critical files such as operating system files, or worse, replace such critical files with bogus copies. This is why Scrip 27 detects, and if so configured prevents pending file rename operations from taking place. However, pending file rename operations also have legitimate uses.

When a Microsoft Windows operating system patch or update is applied, often it needs to replace files currently used and locked by the operating system. These files cannot be replaced while Microsoft Windows is running. Microsoft Windows system updates use pending file rename operations to replace files in use at the time the software update or patch was installed. When the system restarts, these operations (like all pending file rename operations) are executed before the operating system loads the target file(s). On Microsoft Windows NT based systems (NT4, 2000, XP, and Server 2003), a Win32 function is called to schedule such operations. They are entered in the following registry key:

  • HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\Session Manager\PendingFileRenameOperations

For example, when applying the Microsoft Windows 2000 Service Pack 4 update, over 200 pending file rename operations are performed. A handful of those operations involve the deleting of temporary files. The rest replace files that were locked by the operating system at the time the service pack was installed.

On systems running the Microsoft Windows 9x operating system, the system update pending file rename operations are entered in the file wininit.ini, located in the operating system directory, typically C:\Windows. Normally, wininit.ini exists as a zero-byte file. The system update pending file rename operations are written in wininit.ini in the [rename] section.

Once the system update pending file rename operations are executed, the entries in the PendingFileRenameOperations registry key, and in wininit.ini are automatically removed.

Please note that the pending file rename operations are also used by software updates and patches for other Microsoft and third-party applications.

Making changes to areas protected by Scrip 27

When you want to add a program to run at start-up, or allow a pending file rename operation to complete (usually when a new application is installed, or a software update or patch is applied) on one or more systems where Scrip 27's disable or delete configuration options are enabled, you should follow the following steps:

  • Disable the relevant Scrip 27 configuration options, or the Scrip, on one, a subset, or all systems

  • Install the program you want to run at start-up, or add the desired pending file rename operation

  • Re-enable Scrip 27 on the system(s) you had disabled it on. As soon as you do that, Scrip 27 runs and rebuilds the list of the programs configured to run at start-up and the list of pending file rename operations.

Attempt to terminate processes that try to change protected registry items

When the Attempt to terminate processes that try to change protected registry items configuration parameter is enabled, if Scrip 27 is able to identify the process that attempted to make a registry change (and the registry change was rejected or disabled), it will attempt to stop that process if it is included in the List of processes that may be terminated configuration parameter, and is not in the List of processes that will never be terminated configuration parameter.

Attempt to delete processes that try to change protected registry items (after termination)

When the Attempt to delete processes that try to change protected registry items (after termination) configuration parameter is enabled, if Scrip 27 successfully terminates the process, or the process is no longer running (but it would have been terminated if it were) the executable file of the process will be deleted. If the file could not be deleted, Scrip 27 configures the system to delete it upon the next system re-start.

Attempt to quarantine processes that try to change protected registry items (after termination)

When the Attempt to quarantine processes that try to change protected registry items (after termination) configuration parameter is enabled, if Scrip 27 successfully terminates the process, or the process is no longer running (but it would have been terminated if it were) the executable file of the process is moved to a hidden folder called quar located in the ASI client directory. If the original file could not be removed, it is marked for deletion upon re-start (the copy operation to the quarantine almost always succeeds).

List of processes that may be terminated

The List of processes that may be terminated configuration parameter contains a list of full executable paths or individual executable names with optional '*' wildcards defining the processes Scrip 27 is permitted to terminate, quarantine, or delete. Each item in the configuration parameter is entered on a new line.

List of processes that will never be terminated (If no path is specified, the program is allowed to run from any directory)

The List of processes that will never be terminated configuration parameter contains a list of full executable paths or individual executable names with optional '*' wildcards defining the processes Scrip 27 will never take a terminate, quarantine, or delete action on. Each item in this configuration parameter is entered on a new line.

The List of processes that will never be terminated configuration parameter supersedes the List of processes that may be terminated configuration parameter. This means that if you enable one of the configuration parameters that will cause Scrip 27 to terminate, delete, or quarantine a process, the Scrip will first attempt to find a match with a process in the list of processes you entered in the List of processes that will never be terminated configuration parameter. If it finds a match, the Scrip will ignore the contents of the List of processes that may be terminated configuration parameter. The List of processes that will never be terminated configuration parameter takes priority, so a process that is included in both the List of processes that may be terminated, and the List of processes that will never be terminated configuration parameters will never be terminated, quarantined, or deleted.

Please note that if you do not specify the full path for a process you want Scrip 27 to terminate, the Scrip will assume that the process can reside anywhere on the system.

How Scrip 27 matches processes attempting changes with processes in processes-to-be terminated and never-to-be terminated configuration parameters

Suppose the List of processes that may be terminated configuration parameter has the * default entry, meaning that all processes that attempt to make a change to a system's configuration areas protected by Scrip 27 will be terminated, if the Attempt to terminate processes that try to change protected registry items configuration parameter is enabled.

If the process attempting to change a registry key in an area protected by Scrip 27 is D:\regedit.exe, here is how different entries in the List of processes that will never be terminated configuration parameter would compare resulting in a match preventing D:\regedit.exe from being terminated, or not in which case D:\regedit.exe would be terminated:

  • D:\regedit.exe – Matches, D:\regedit.exe would not be terminated.

  • regedit.exe – Does not match, D:\regedit.exe would be terminated.

  • *regedit.exe – Matches, D:\regedit.exe would not be terminated.

  • *\regedit.exe – Matches, D:\regedit.exe would not be terminated.

Since the full path name (D:\regedit.exe) is used for the comparison, the second entry would not match. We would recommend using the fourth entry to prevent processes such as D:\virusregedit.exe from matching an entry in the List of processes that will never be terminated configuration parameter if you want to use process names as entries in the List of processes that will never be terminated configuration parameter.

Permitted items function

How it works

When it detects an attempt to add, delete, or change an item to a system's start-up environment, Scrip 27 compares the start-up item it detects with the content of the Permitted items configuration parameter. If it finds a match, it will allow the addition, deletion, or change of the start-up item. Otherwise, it will disable or reject the addition, deletion, or change, depending on how it is configured.

Scrip 27 processes the list of items in the Permitted items configuration parameter sequentially. It tests an item for one of the following potential matches:

  1. Checks if the start-up item matches exactly an item in the Permitted items configuration parameter.

  2. The start-up item is a registry entry and its registry branch matches a RegBranch (registry branch) entry in the Permitted items configuration parameter.

  3. The start-up item is a registry entry and its key variable name, and/or key variable value match a RegValString (registry key variable value sub-string) entry

  4. A portion of the start-up item matches a Keyword entry in the Permitted items configuration parameter

Permitted items

You can add entries to the Permitted items configuration parameter manually, and by using the Populate “Permitted items” list now action. Permitted items configuration parameter entries can be of the following type:

Registry key

RegKey,<user name>,<key name>,<key variable name>,<key variable value>

Registry key variable value sub-string

RegValString,<user name>,<key name>,<key variable name>,<key variable value>

Registry branch

RegBranch,<any portion of a registry key path except for registry key <key variable name> and <key variable value>>

Service

Service,<key name>,<display name>,<image path>

Keyword

Keyword,<keyword value>

Start folder

StartFolder,<User name>,<Start-up folder item path>,<Start-up folder item>

Ini file

IniFile,<User name>,<.ini file (either win.ini or system.ini) path>,<.ini file section>,<.ini file variable name>,<.ini file variable value>



Please note that in any of the above entries, there are no spaces before or after commas.

Populate "Permitted items" list now

If you click on the Populate “Permitted items” list now Execute button, Scrip 27 will populate the Permitted items configuration parameter with the content of the Currently enabled startup items on this machine – known Scrip 233 configuration parameter.

Please note that taking the Populate “Permitted items” list now action will populate the Permitted items configuration parameter with items that will be used by the permitted items function for exact matches. In other words, Scrip 27 will allow deletion, or changes only of items matching exactly entries in the Currently enabled startup items on this machine – known Scrip 233 configuration parameter.

Taking the Populate “Permitted items” list now action will not overwrite the content of the Permitted items configuration parameter.

Since the Permitted Items configuration parameter can be assigned to one system, a group of systems, or an entire site, when populate the Permitted Items configuration parameter using the Populate Permitted items Execute button, its content can be propagated automatically to all systems at a site, or a group of systems.

Standard interface

User interface enabled

The User interface enabled configuration option lets you enable or disable the user interface (end-user dialog box). When the user interface is enabled, and an attempt to add a start-up item, an attempted pending file rename operation is detected, or a system update operation is detected, the attempt is reported in a dialog box. The user is given the option to delete the start-up item (or pending file rename operation), disable it, or cancel (leave the item alone). The user's choice overrides the Do not allow the addition of startup items, Do not allow the removal of startup items, Do not allow the addition of system update operations, Disable rather than delete added startup items, and Disable rather than delete added system update operations configuration options, even if they are enabled. By default, this option is disabled.

If the end-user does not take any action when the user interface dialog box appears, after 60 seconds two events can occur:

  • If some or all of the Do not allow the addition of startup items, Do not allow the removal of startup items, Do not allow the addition of system update operations, Disable rather than delete added startup items, and Disable rather than delete added system update operations configuration options are enabled, Scrip 27 will take action accordingly, with the Disable configuration options overriding the Do not allow options, if both are enabled.

  • If none of the Disable or Do not allow options are enabled, Scrip 27 will take no action.

Do not allow the addition of startup items

Enabling the Do not allow the addition of startup items configuration option lets you delete start-up items whose attempted addition is detected by Scrip 27 rather than just disable them. By default, this option is disabled.

Do not allow the removal of startup items

Enabling the Do not allow the removal of startup items configuration option will prevent the deletion of items from a system's start-up environment, e.g. entries in the Run registry key. By default, this option is disabled.

Do not allow the addition of system update operations

Enabling the Do not allow the addition of system update operations configuration option will prevent pending file rename operations from adding entries (e.g. to delete, move, or copy certain files) to a system's start-up environment (e.g. the file wininit.ini). By default, this option is disabled.

Please note that when installing a Microsoft Windows software update or patch, the Do not allow the addition of system update operations configuration option must be disabled until after system restart. If it is not disabled, the system may end up in a unknown, unstable state.

Disable rather than reject added startup items

If the Disable rather than delete added startup items configuration option is enabled, Scrip 27 will disable start-up items whose attempted addition is detected by Scrip 27. If this parameter is enabled it overrides the Do not allow the addition of startup items configuration option. By default, this option is disabled.

Please note that the Disable rather than delete added startup items configuration option has no effect unless the Do not allow the addition of startup items configuration option is enabled.

Disable rather than reject added system update operations

If the Disable rather than delete added system update operations configuration option is enabled, Scrip 27 will disable entries added to a system's start-up environment (e.g. wininit.ini) by pending file rename operations. If this parameter is enabled it overrides the Do not allow the addition of system update operations parameter, even if it is enabled. By default, this option is disabled.

The Disable rather than delete added system update operations configuration option has not effect unless the Do not allow the addition of system update operations configuration option is enabled.

Please note that when installing a Microsoft Windows software update or patch, the Disable rather than delete added system update operations configuration option must be disabled until after system restart. If it is not disabled, the system may end up in a unknown, unstable state.

Note on prioritization of actions

If both the Disable rather than delete added startup items and Do not allow the addition of startup items options are enabled and the User interface enabled option is not, the Disable rather than delete added startup items option overrides the Do not allow the addition of startup items option. Similarly, if both the Disable rather than delete added system update operations, and Do not allow the addition of system update operations options are enabled and the User interface enabled option is not, the Disable rather than delete added system update operations option overrides the Do not allow the addition of system update operations option.

if the User interface enabled option is disabled, Scrip 27 immediately executes the default action. if the User interface enabled option is enabled, Scrip 27 executes the the default action only if the end-user does not respond to the user interface dialog box request within 60 seconds. Otherwise, the end-user's selection is executed.

Advanced interface

The advanced interface lets you select the action the Scrip should take depending on the area being attacked. The possible actions are:

  • 1 - Scrip 27 will accept changes, taking no action. Changes will still be logged.

  • 2 - Scrip 27 will reject changes, restoring original conditions.

  • 3 - Scrip 27 will reject changes, and if possible, will allow for rejected changes to be re-enabled at a later time.

  • 4 - Scrip 27 will display user interface and take action 1 if dialog box times out.

  • 5 - Scrip 27 will display user interface and take action 2 if dialog box times out.

  • 6 - Scrip 27 will display user interface and take action 3 if dialog box times out.
