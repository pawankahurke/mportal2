Impersonation Support

On NT based systems (MS Windows NT4, 2000, XP, and Server 2003) the ASI client is logged in as local system administrator. This means that it cannot access shared resources on the local network.

In order to enable ASI client's access to shared resources on the local network, we have implemented impersonation. Impersonation lets the ASI client log onto the system where the shared resource is located as a user known by that system enabling it to access that shared resource.

You should enter the known user id and password in Scrip 43, as described below. With this information, the ASI client will be able to access shared resources on the local network as follows:

For Workgroups

To MS Windows 9x

To MS Windows NT (NT4, 2000, XP, and Server 2003)

From MS Windows 9x

Not supported

Not supported

From MS Windows NT (NT4, 2000, XP, and Server 2003)

Supported
User account needs to exist on from system

Supported
User account needs to exist both on from and to systems
For Domains

To MS Windows NT (NT4, 2000, XP, and Server 2003)

From MS Windows NT (NT4, 2000, XP, and Server 2003)

Supported
User account needs to exist on domain controller

Impersonation enabled

When the Impersonation enabled parameter is enabled, using the impersonation user id and password (and domain if necessary) you entered in Impersonation Username, Impersonation Password and Confirmation, and Impersonation Domain parameters in the Scrip 43 configuration page, Scrip 100 can retrieve files that reside on shared resources on other systems on the network, accessed either as mapped drives(e.g. x:\shared folder), or using UNC (Universal Naming Convention) syntax (e.g. \\server\\shared folder).

Send to default email group

When you enable the Send to default email group parameter, Scrip 100 will send the file(s) you wish to retrieve to the e-mail address(es) entered in the Email address for administration group parameter in the Scrip 43 (HandsFree Client Tools) configuration.

The default value of the Email address for administration group parameter is support@handsfreenetworks.com. You can change this value easily in the Scrip 43 configuration page. You can enter multiple e-mail addresses, each separated by a comma, as the value of parameter Email address for administration group.

This will give you the ability to identify one or more email addresses that can be used globally by any Scrip in one place. When one or more of these e-mail addresses changes you will have to make that change in only one place rather than having to modify all the Scrips that use that e-mail address.

Email address group definitions

Email address group definitions parameter entries have the following format:

  • Name of the email group, followed by a comma

  • Email address(es) separated by commas

You can have as many email groups defined as you want with each group entered on a new line.

Instead of having file(s) sent to an e-mail box, you can have the ASI client include the file(s)in the log posted by Scrip 100 on your log server. You do this by entering in your file retrieval command line LogServer as the name of the e-mail group you want files sent to. Please note that you do not need to create an e-mail group called LogServer. It's built in the ASI client.

If, based on your selection criteria, Scrip 100 retrieves more than one file, and the Log each file in separate event logs configuration parameter is enabled, it will generate one log for each file retrieved.

When sent to e-mail group LogServer, the content of the file retrieved is stored in field text3 in the event database on your log server. Please keep in mind that the size of a file incorporated in a log sent to the log server via the LogServer e-mail group is limited to 64 KBytes.

Keyword group definitions

Keyword group definitions configuration parameter entries have the following format:

  • Name of the Keyword group, followed by a comma

  • keyword(s) separated by commas

You can have as many keyword groups defined as you want with each group entered on a new line.

File distribution. File location group definitions

In configuration parameter File distribution. File location group definitions you define each location where you might want to distribute files.

Entries in File distribution. File location group definitions, called location groups, are on separate lines. Each entry has two components separated by commas, with no spaces:

  • The first is the name you choose to give the location, e.g. hosts

  • The second is the actual location on each system where you want Scrip 100 to place the file(s) to be distributed, e.g. c:\windows

Please note that a location group can contain multiple locations. For example,

everywhere,c:\windows\application data,c:\program files\common files,d:\program files\common files

URL group definitions

In configuration parameter URL group definitions you define each location where you might want to send files from the system where Scrip 100 runs via a file transfer method different from e-mail or, because of file size and other constraints, when you may not want or be able to include the file in an event log posted on the ASI server.

Currently, we only support file transfer via FTP.

Configuration parameter URL group definitions may have multiple entries, each on a separate line. A configuration parameter URL group definitions entry has two components separated by commas with no spaces:

  • The first is the name you choose to give the URL group, e.g. disasterbackup

  • The second is URL for the file transfer, e.g. ftp://nanoheal.org/ - The URL must not include a file name.

Please note that a location group can contain multiple URLs. For example:

  • allbackups,ftp://nanoheal.org,ftp://hfndev.com,ftp://corporateink.dnsalias.com

If you authentication is required in order to access the target server, you need to embedded it in the URL. For example:

  • securebackup,ftp://upload:abcdefgh@nanoheal.org/

File distribution. Overwrite existing file(s)

When the File distribution. Overwrite existing file(s) configuration parameter is enabled, the file(s) to be distributed will overwrite file(s) with the same name.

Retrieve files by date

When you enable the Retrieve files by date configuration parameter, Scrip 100 will only retrieve files, matching the mask in the File retrieval command lines configuration parameter, whose last modified date falls in the time interval starting from the date you enter in the What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date) configuration parameter going backward for a number of days equal to the value you enter in the How many days back from the latest date do you want to retrieve files for? Configuration parameter.

What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date)

Please note that when you leave the What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date) configuration parameter blank, Scrip 100 will set the value of this parameter to the date on which it runs (what we refer to as the current date.

How many days back from the latest date do you want to retrieve files for?

If you set the value of the How many days back from the latest date do you want to retrieve files for? configuration parameter to 0, Scrip 100 will only retrieve files whose last modified date is equal to the date you entered in the What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date) parameter.

Log each file in separate event logs

When you select LogServer as the retrieved file destination, and the Log each file in separate event logs configuration parameter is enabled, Scrip 100 will post N + 1 event logs, N, each containing one of the N files it retrieves, and one containing the Scrip's configuration at the time of execution, and a report of the actions taken. In each of the N event logs, each containing a file retrieved:

  • Field text1 will consist of: Attachment #X (X = 1 to the number of files retrieved by Scrip 100), followed by the string path identifying the file contained in the log.

  • Field text3 will consist of the content of one of the files retrieved by Scrip 100

File distribution and retrieval command line(s)

The File retrieval command line(s) configuration parameter contains one or more command lines separated by a carriage return, or new line feed.

The components of a file retrieval command line are separated by a comma, with no spaces. Each command line has the following format:

  • 0 (disable) or 1 (enable) - this makes it possible to temporarily stop retrieving certain file(s) without deleting the corresponding file retrieval command line from the list.

  • The name of the file(s) you would like Scrip 100 to retrieve and send. You can use the standard Microsoft MS-DOS wild card characters ? and * in both the file name and extension. Please note that you need to enter the complete path for the file(s) you want retrieved.

  • the name of the email group, or location, you want to send the file(s) to. Please note that each file entry, potentially containing multiple files, can only go to one email group.
    When using the Scrip 100 file distribution function, file(s) will be distributed to the locations listed in the location group entry in parameter File distribution. File location group definitions.

  • 1 - if you want the file(s) to be deleted after sending is completed, 0 - if you don't want the file(s) to be deleted.

  • 1 - if you want Scrip 100 to perform a keyword search to decide whether to send the file(s) in the file entry, 0 - if you do not want a keyword search performed

  • The name of the keyword search group you want Scrip 100 to use if you entered 1 in the preceding field. You can enter only one keyword group per command line.

  • 1 – if you want the current setting of the File distribution. Overwrite existing file(s) configuration parameter to apply to a command line, 0 - if you do not.

  • 1 – if you want the current setting of the Retrieve files by date configuration parameter to apply to a command line, 0 - if you do not.

  • The date (MM/DD or MM/DD/YY) value which you want to use instead of the value entered in the What is the latest date you want to retrieve files by? (MM/DD or MM/DD/YY, leave blank if you want current date) configuration parameter

  • The number of days back which you want to use instead of the value entered in the How many days back from the latest date do you want to retrieve files for? configuration parameter

  • 1 – if you want the current setting of the Log each file in separate event logs configuration parameter to apply to a command line, 0 - if you do not.

If in a file distribution or retrieval command line you do not want to enter a value for a parameter located between two parameters you want to use, you should simply enter a comma with no spaces in the place in the command line where the parameter you do not want to use is located. If the parameters you do not want to use are at the end of a command line, you can ignore them. Please see below for some examples.

If you choose to send files retrieved by Scrip 100 only to the default email group, the file retrieval command line should contain two commas after the entry for the name of the file(s) you would like the Scrip to retrieve and send. For example:

  • 1,\\server5\e-drive\HandsFree Networks\*.txt,,,0 - Will send files matching the mask *.txt to the default email group without deleting them, and without searching for keywords

In the file retrieval command line, if you don't specify a value for file deletion or keyword search group, Scrip 100 will use the default values of 0, and 0, respectively.

Scrip 100 can retrieve and distribute files located on local network accessible shared resources, FTP, and Web servers (both HTTP and HTTPS).

Following are examples of file retrieval command lines and how the Scrip would process them:

  • 1,\\server5\e-drive\HandsFree Networks\*.txt,me,,0 - Will send files matching the mask *.txt to "me" without deleting them, and without searching for keywords

  • 1,\\server5\e-drive\HandsFree Networks\*.txt,me - Will send files matching the mask *.txt to "me" without deleting them, and without searching for keywords

  • 1,\\server5\e-drive\HandsFree Networks\*.txt,me,1,ErrorLogs - Will send files matching the mask *.txt to "me", delete the files and NOT look for any keywords

  • 1,\\server5\e-drive\HandsFree Networks\*.txt,me,, - Will send files matching the mask *.txt to "me" without deleting them, and without searching for keywords

  • 1,ftp://texaco.tellink.net/pub/cfg.ini,windows - Will distribute file cfg.ini to location(s) specified in the location group windows downloading it from the ftp server at texaco.tellink.net.

  • 1,http://texaco.tellink.net/pub/cfg.ini,windows - Will distribute file cfg.ini to location(s) specified in the location group windows downloading it from the Web server at texaco.tellink.net. Please note that the file cfg.ini could also have been downloaded from a secure Web server (using HTTPS).

Please note the following limitations in the use of the Scrip 100 command line options:

  • The file(s) delete-after-sending option is not applicable to the file retrieval function when the file(s) being retrieved/distributed is/are on an FTP or Web server (via HTTP/HTTPS).

  • The file(s) delete-after-sending option is not applicable to the file distribution function.

  • Wildcards are not supported when retrieving/distributing files from an FTP or Web server (via HTTP/HTTPS).

  • The keyword matching option is not applicable to the file distribution function.

Distribute / Retrieve file(s) now

Clicking on the Distribute / Retrieve file(s) now Execute button, will trigger Scrip 100 execution on demand, even if it's not enabled. In Scrip 100, the Execute button can be used to execute the Scrip on the local system or on all systems at a site.

Schedule

The Schedule parameter has a command line with eight fields, each entered on a new line.

The table below describes each field in detail.

Field Number

Field Name

Field Values and Description

1

Minutes

0-59; a value of * means that the Scrip should run every minute

2

Hours

0-23; a value of * means that the Scrip should run every hour

3

Day of the Month

1-31, depending on the month; a value of * means that the Scrip should be run every day.

4

Month

1-12; a value of * means that the Scrip should be run every month.

5

Day of the Week

0, Sunday - 6, Saturday; a value of * means that the Scrip should be run every day of the week.

6

Minute Randomizer

N; the size of the time interval in minutes over which the Scrip should be randomly executed. We multiply N*60, then calculate a random number X between 1 and N*60, and run the Scrip X seconds after its scheduled time.

7

Conditional Execution

N; N=1 means that if the Scrip is past due, it will be run immediately, N=2 means that If the Scrip is past due, it will be run at the next scheduled execution time. If the value is set to 2, the actual Scrip execution time will be within 3 minutes (either way) of the scheduled time. This is necessary to prevent the case where a process will not be executed if it is a little slow to start, or if the scheduler starts it a few seconds too early.

8

Failed Run Limit

N, where N is the number of times that Scrip execution is allowed to fail before notification is sent to the support provider; N=0, the Scrip will NEVER report a failure, N=1, the Scrip will report EVERY failure, N=2, the Scrip will report if it failed twice in a row, and so on.



The Minutes, Hours, Day of the Month, Month, and Day of the Week fields can have multiple values and ranges of values separating them with commas.

For example, a 1,3,5 entry for the Minutes parameter would mean that a Scrip would be executed at minutes 1, 3, and 5.

A 10-14 entry for the Hours parameter would mean that a Scrip would be executed starting at 10 AM and ending at 2 PM.

Multiple value and range entries can be used together separated by commas.
