Site

Site contains the name of the site used for all ASI logs posted and activties performed by the ASI client at a site.

Please note that the first time an ASI client from the site contacts the ASI event management facility on the ASI server, it will look for an existing site name that matches the one contained in Site. If it does not find one, the ASI event management facility will create a site name that matches the one contained in Site, and will store all event logs using that newly created site name.

The same sequence of events will occur the first time an ASI client from the site contacts the ASI asset management and site management facilities.

We strongly advise against changing the name of a site by changing the content of the Site parameter in Scrip 43 because it would be difficult for you to uniformity of site name particularly where systems belonging to a site are distributed across multiple local networks over a widely distributed area, or simply consist of remotely connected and/or mobile systems.

Additional complexity comes from having to make sure that when you change the value of the Site parameter, it is set to global, or local, as needed, otherwise you would end up with systems reporting incorrect site information.

If you want to change the name of a site, the easiest and simplest way to do it is to change the site's name in its profile on the ASI installation management facility. Doing it this way, avoids all the issues described in the two paragraphs above.

User name for configuration authentication

In the User name for configuration authentication parameter, you enter the user name used to access the ASI client directly by logging onto it via a browser session from any computer with direct access to the network where systems where the ASI client is installed are located.

Please note that the system from which you access the ASI client does not have to have the client installed.

Configuration password

The Configuration password configuration parameter lets you change the password for directly accessing the ASI client.

Because the Configuration password parameter can be global or local, you can effectively and simply control access to Scrip configuration conforming to your organization's security and control policies.

The configuration password is used in conjunction with the user name entered in the User name for configuration authentication parameter.

Machine identifier type

When Machine identifier type is set to 0, the ASI client will use a system's NetBIOS name as the value of field machine in event logs sent either to the server(s) listed in ASI event management server URL list, or via e-mail. If Machine identifier type is set to 1, The ASI client will use the Universally Unique Identifier (UUID) it generates upon first-time installation as the value of field machine. When Machine identifier type is set to 2, the ASI client will append the value you enter in the Friendly name when machine identifier type = 2 configuration parameter to the system's NetBIOS name.

By default, Machine identifier type is set to 0.

Friendly name when machine identifier type = 2

The Friendly name when machine identifier type = 2 configuration parameter lets you easily modify the NetBIOS name of a system by appending a label of your choice right after it. For example, if the NetBIOS name of a system is node1, and the value you enter in the Friendly name when machine identifier type = 2 configuration parameter is John Smith, then the name of the system used by the ASI client and all Scrips, and on the ASI server will be node1John Smith.

If the value of the Machine identifier type is set to 2, and the Friendly name when machine identifier type = 2 configuration parameter is blank, it's as if the value Machine identifier type were set to 0. No friendly name will be appended to a system's Net BIOS name.

Because the Friendly name when machine identifier type = 2 configuration parameter can be either local or global, before setting its value it's very important that you set it to either global or local, depending on what you want to do.

The value of the Friendly name when machine identifier type = 2 configuration parameter is stored in a registry key. Please note that this parameter does not change or affect in any way a system's NetBIOS name.

Proxy URL

If the site your are configuring the ASI client for uses a proxy server, you enter the proxy server information in the Proxy URL parameter.

An entry in Proxy URL has the following format:

http://[user name]:[user password]@[proxy server name]:[proxy port]

If user authentication is not required, you do not need [user name] and [user password].

Domain for proxy authenticaton

If the site you are configuring uses a proxy server that requires NTLM (NT LAN Manager) authentication, then you may need to enter the Domain for proxy authentication parameter. For example, Microsoft ISA Proxy requires NTLM authentication by default.

The Domain for proxy authenticaton parameter is the NT domain for which the proxy user name and password specified in the Proxy URL parameter are valid. If you only have one NT domain, use that domain. If you are using workgroups rather than NT domains, you do not need to fill in this parameter.

ASI event management server URL list

In ASI event management server URL list you should enter the URL of the server(s), ASI clients at the site should log events to. If you have back-up event management server(s) that ASI clients can use to log events to in case the primary event management server is not accessible, ASI event management server URL list will consist of the URLs of the servers each on a different line, with no blank lines between entries.

Please note that for each server, you will have to enter the full HTTPS or HTTP command line depending on whether you want the ASI client-to-server transactions to be SSL encrypted or not. For example:

https://asi.myevents.com

Management server fail-over scheme

You can use the Management server fail-over scheme configuration parameter to decide which strategy the ASI client should use when there are multiple entries in any of the management server URL list configuration parameters. Please note that the Management server fail-over scheme configuration parameter applies to ALL management server URL list configuration parameters.

If the Management server fail-over scheme configuration parameter is set to 1, its default value, the ASI client always tries to log to the server listed first in the Management server fail-over scheme configuration parameter. If it is not available, it will try to log to the second server on the list, and so on, until it finds a server which is available. If Management server fail-over scheme is set to 2, the ASI client logs to the last server it was able to reach.

Use HTTP/HTTPS for event logging

When the Use HTTP/HTTPS for event logging option is enabled, the ASI client will send logs to the server(s) listed in ASI event management server URL list via HTTP or HTTPS depending on the URL entered in ASI event management server URL list.

Use HTTP/HTTPS for event logging is enabled by default. if both Use HTTP/HTTPS for event logging and Use email for event logging are enabled, the former overrides the latter.

Use email for event logging

When the Use email for event logging option is enabled, and the Use HTTP/HTTPS for event logging option is disabled, the ASI client will send logs to the e-mail addresse(s) listed in the Email address for administration group, Email address for technical support group, Email address for engineering group, or Return address for bounced email messages parameters, depending on the Scrip that produced the log. Please note that you do not have the option to change which Scrip send logs to which e-mail address.

Site domain for email logging

Site domain for email logging is the domain used to generate the return email addresses when the ASI client sends logs via e-mail. For example, if the site domain is company.com and the machine name is nodeN, the return address would be nodeN@company.com.

Destination e-mail addresses for e-mail logging

Email address for administration group, Email address for technical support group, and Email address for engineering group, are the addresses to which event logs are sent when the Use email for event logging option is enabled.

Return address for bounced email messages

When ASI client logs sent via e-mail do not reach any one of the addresses entered in the Email address for administration group, Email address for technical support group, Email address for engineering group, and Return address for bounced email messages parameters, they will be returned to the e-mail address you enter in the Return address for bounced email messages parameter.

SMTP server for email logging

When sending logs via e-mail, the ASI client will use the SMTP server whose name or IP address you enter in SMTP server for email logging.

If SMTP server for email logging is empty, The ASI client first tries to retrieve the name of the authoritative mail server for the domain by using DNS. If that fails, the ASI client tries to use the sub-net's gateways as mail servers.

Window hooks enabled

The Window hooks enabled option lets you enable or disable detection of Microsoft Windows messages. It is enabled by default and you should disable it only under exceptional circumstances. When it is disabled, as significant portion of the ASI client's event detection capabilities would not be operational.

Disabling the Window hooks enabled option will affect the following Scrips:

  • Scrip 38 will no longer detect faults. However, the ASI client will still trap its own faults.

  • There are a few Scrips that close dialog boxes after running maintenance tasks. These will no longer close those dialog boxes. They are:

    • Scrip 10 -- closes "ScanDisk Results" dialog after Scandisk completes execution.

    • Scrip 13 -- closes "Please Read" and "Norton Anti Virus" dialogs after running Norton Anti Virus.

    • Scrip 20 -- closes dialog after defrag.

  • The following Scrips won't be triggered:

    • Scrip 68 -- detects and corrects time/date change

    • Scrip 94 -- detects and logs dialog boxes

    • Scrips 46, 47, 48, 49 -- detect and disable error, warning, information and question dialog boxes (These Scrips are disabled by default)

    • Scrip 192 -- controls execution of programs on a system

  • The Scrips listed below also won't be triggered:

    • Scrip 14 -- detects "Internet Accounts" dialog

    • Scrip 16 -- detects "Delete history" IE dialog

    • Scrip 21 -- detects "Confirm File Delete" and "Confirm Folder Delete" dialogs for recycle bin

    • Scrip 25 -- detects dialogs that are created by the client itself

    • Scrip 76 -- detects Netscape preferences change

    • Scrip 79 -- detects Eudora preferences change

    • Scrip 80 -- detects Pegasus preferences change

Enable client access from the network if system firewall is active

The Enable client access from the network if system firewall is active configuration parameter applies only to systems running the Microsoft Windows XP operating system. When the Microsoft Windows XP native firewall is enabled, enabling the Enable client access from the network if system firewall is active configuration parameter lets ASI clients continue to be able to communicate with each other via port 2721.

Enable Scrip time-out checking

When the Enable Scrip time-out checking configuration parameter is enabled, an execution time threshold is set for selected Scrips. This threshold varies with each Scrip. If one of these Scrips' execution time exceeds its threshold, the ASI client restarts.

Dump failed logging attempts to error log file

When Dump failed logging attempts to error log file is enabled, The ASI client will write all HTTP/HTTPS errors detected when sending logs to the ASI server to a log file called errlog.txt that resides in the ASI client directory.

When the ASI client detects the presence of errlog.txt in its directory, it automatically sends it to the ASI server. The information contained in errlog.txt is very helpful in diagnosing HTTP/HTTPS communication problems. errlog.txt is in plain text format. The data it contains is traceback information and HTML messages the ASI server generates when an error occurs.

Please note that you should not enabled the Dump failed logging attempts to error log file parameter unless necessary for diagnosing a problem. Otherwise, any failure to log events to the ASI server will trigger the dump, even when the failure is not caused by a problem but to conditions such as a system not being connected to the Internet by choice.

Put window text into dialog box event logs

When Put dialog box content into dialog box event logs is disabled, the ASI client will include only the dialog box title in the dialog box event logs. This parameter is enabled by default.

Password for local user interface

The Password for local user interface configuration parameter lets you manage and control the password for accessing the local ASI client user interface. By default The Password for local user interface configuration parameter, is empty. In this case, the ASI client will give access to the local user interface using dilbert as the password.

Impersonation Support

In order to enable ASI client's access to shared resources on the local network, we have implemented impersonation. Impersonation lets the ASI client log onto the system where the shared resource is located as a user known by that system enabling it to access that shared resource.

You should enter the known user id and password in Scrip 43, as described below. With this information, the ASI client will be able to access shared resources on the local network as follows:

Impersonation username

The Impersonation username contains the name of the user that can be used by Scrips that need to access shared resources on NT based systems (systems running the Microsoft Windows NT4, 2000, XP, or Server 2003 operating system).

Impersonation password

The Impersonation password configuration parameter holds the password that in conjunction with the Impersonation username lets Scrips access shared resources on other systems on the network, either as mapped drives(e.g. x:\shared folder), or using UNC (Universal Naming Convention) syntax (e.g. \\server\\shared folder).

Impersonation domain

The Impersonation domain contains the name of the domain that can be used, if necessary, by Scrips that need to access shared resources on other systems on the network either as mapped drives(e.g. x:\shared folder), or using UNC (Universal Naming Convention) syntax (e.g. \\server\\shared folder).

Maximum number of Scrip errors within threshold time before disabling Scrip

When the number of errors that occur during a Scrip's execution within the time interval set in the Threshold time for Scrip errors (seconds) configuration parameter, exceeds the value set in the Maximum number of Scrip errors within threshold time before disabling Scrip configuration parameter, a Scrip's execution will terminate. The value of this configuration parameter applies to all Scrips. By default it is set to 3 errors.

Threshold time for Scrip errors (seconds

In the Threshold time for Scrip errors (seconds) configuration parameter you set the interval of time during which the number of errors that occur during any Scrip's execution cannot exceed the number you set in the Maximum number of Scrip errors within threshold time before disabling Scrip configuration parameter. The value of this configuration parameter applies to all Scrips. By default it is set to 60 seconds.

Scrip timeout when unspecified (seconds

In the Scrip timeout when unspecified(seconds configuration parameter you set the interval of time after which execution of a Scrip (including both queue and execution time) is terminated. When this happens, a traceback that includes a stack trace for the timed-out Scrip is generated and uploaded to the ASI server, and the ASI client will restart. A few Scrips set their own time limit. We also have a mechanism for Scrips like Scrip 23 (Send Queued Event Logs) to extend their time limit as long as they know that they are doing useful work (this is a common technique called a "watchdog timer").

Execution of any Scrip that doesn't explicitly set its own time limit is terminated when the time specified in the Scrip timeout when unspecified (seconds configuration parameter is exceeded. The default value for this configuration parameter is set at 12 hours (43200 seconds). If you set it to zero, then Scrips that don't set their own time limit will never time out (in other words, zero means infinite).

The Scrip timeout when unspecified (seconds) configuration parameter can also be used to terminate the execution of Scrips that may be stuck in the run queue. If a Scrip is stuck in the run queue (which you can see using a Scrip dump), then you can set the value of the Scrip timeout when unspecified (seconds) configuration parameter to something less than the time the Scrip you want to take out of the run queue has been in the queue. The next time the ASI client compares the value of the Scrip timeout when unspecified (seconds) configuration parameter with the time the Scrip has been in the queue, it will take the Scrip out of the queue. A traceback that includes a stack trace for the timed-out Scrip will be generated and uploaded to the ASI server, and the ASI client will restart. The stack trace will show us where the Scrip was stuck. Of course, you usually want to set the default time limit back up once the client restarts. This entire operation can be performed remotely.

McAFee VS version override

If we determine that the current scrip is compatible with a newer release of McAfee, then by setting the value of the McAfee VS version override configuration parameter equal to the current major build number for the new release, Scrips 90 (McAfee Virus Definition Management), and 175 (McAfee VirusScan Execution) will continue to work without any changes.

Please note that the value of the McAfee VS version override configuration parameter should be changed only by HandsFree Networks personnel.

Logs Overview

  • Comm dump: A list a recent network communications between clients and or the server.

  • Alist dump: A list of recent calls to add and remove machines from the client machine list.

  • Sync dump: A list of sync objects in use. Used in client resource management.

  • Scrip dump: A list of recently executed scrips, scrips currently being executed, and scrips waiting to be run.

  • Timer dump: A list of scrips in the timer queue scheduled to be run.

Site registration code flush (only change when instructed to do so by technical support)

You should not use the Site registration code flush (only change when instructed to do so by technical support) configuration parameter without first consulting HandsFree Networks.

Here is how the Site registration code flush (only change when instructed to do so by technical support) configuration parameter works. You fill in the new site registration code and submit the change. This causes Scrip 43 to run. It copies the site registration code into a special registry key, clears the Site registration code flush (only change when instructed to do so by technical support) configuration parameter, and restarts the ASI client. When the ASI client starts, it copies that registry key into the site registration code and flushes the password list. This causes the ASI client to behave as it always does on a site registration code change, but prevents it from communicating with other ASI clients.

the Site registration code flush (only change when instructed to do so by technical support) configuration parameter needs to be local-only because it wouldn't propagate anyway, or at least it wouldn't propagate reliably, because the ASI client shuts down right after it changes. Also, the whole point is that this change is supposed to isolate the system where Scrip 43 is run in this mode from the others at the site, so you wouldn't want it to propagate.

You can use the Site registration code flush (only change when instructed to do so by technical support) configuration parameter to just clear the password list by setting the new value of the site registration code to be the same as the current site registration code.

Note that the Site registration code flush (only change when instructed to do so by technical support) configuration parameter also affects how the /ID command line parameter in the ASI client installation executable works. If you use the /ID command-line parameter when installing the ASI client to set the site registration code, if the ASI client is being run on a system where the ASI client is already installed, it will replace the current site registration code, and flush the password cache. So, the /ID command line parameter can also be used to isolate a system from the others.

Generate new HandsFree Client UUID

Under certain circumstances, you could end up with multiple systems have the same UUID (unique universal identifier) generated by the ASI client upon installation. For example, if you include the ASI client in a standard image, every system where you install that image will have the same UUID. This causes a number of problems including degraded ASI server performance. Should it arise, to correct this situation you can use the Generate new HandsFree Client UUID function. Clicking on the Generate new HandsFree Client UUID Execute button will archive and delete a system's current UUID, and generate a new one.

Clear configuration database and restart

Clicking on the Execute button to the right of the Clear configuration database and restart label will delete the ASI client and Scrip configuration database, and re-start the ASI client. Upon restarting, the ASI client will automatically re-create the ASI client and Scrip configuration database and re-populates it with configuration values drawn from other copies of the ASI client and configuration database either on the sub-net or on the ASI server.

Send error log file

Clicking on the Send error log file button will immediately send the Errlog.txt file found in the ASI client directory to the log server for the local site, or e-mail it to the designated recipient if e-mail logging is enabled.

Send mail

When e-mail logging is set to true, clicking on the Send mail button will immediately sen any e-mail logs found in the email queue to the designated recipient.

ASI client uninstall

Clicking on the Execute button to the right of the ASI client uninstall label will remove the ASI client from the system where the Scrip 43 configuration page is being accessed.

ASI client uninstall - silent

Clicking on the Execute button to the right of the ASI client uninstall - silent label will remove the ASI client from the system where the Scrip 43 configuration page is being accessed. This uninstall mode is called silent because no dialog boxes will appear on the screen during the uninstall process.
