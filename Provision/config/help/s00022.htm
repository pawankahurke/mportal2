Scrip 22 Configuration

Scrip 22 Default Configuration Parameters

  • Scrip enabled

  • Create event log for asset log failures disabled

  • Initial delay in seconds before retrying failed asset logs 60 seconds

  • Maximum delay in seconds before retrying failed asset logs 3600 seconds

  • Initial limit on random backoff in seconds 900 seconds

  • Maximum limit on random backoff in seconds 7200 seconds

  • Maximum number of asset logs to queue 100

  • Send asset logs now

Scrip 22 manages the process of retrying to send asset logs generated by Scrip 61 (System Survey) to the ASI log server when that process initially fails.

Create event log for asset log failures

If you enable the Create event log for asset log failures configuration parameter, asset log failures will be logged as Scrip 22 event logs on the ASI server. By default, this configuration parameter is disabled.



There are two reasons why an asset logging operation can fail, forcing the client to re-try the operation. These cases are somewhat different, so the ASI client handles them differently:

If the ASI client cannot reach the server, Scrip 22 will use the Initial delay in seconds before retrying failed asset logs and the Maximum delay in seconds before retrying failed asset logs configuration parameters described below to manage the asset log transmission process.

In this case, there is no scarce resource. Instead, it is just a transient failure (Note that we are ignoring the case where the failure is caused by congestion.) Often the failures are short-lived, so it is worthwhile to retry quickly. However, they are sometimes lengthy failures, so it is a bad idea to consume resources by continuously retrying. The best adaptive algorithm in this case is a fixed binary backoff, where the retry delay starts at a short interval, and the interval is doubled every retry, up to a maximum retry interval.

Initial delay in seconds before retrying failed asset logs

The Initial delay in seconds before retrying failed asset logs configuration parameter contains the starting value for asset log transmission retry interval. In other words, the first time Scrip 22 cannot reach the ASI server, it will wait the number of seconds specified in the Initial delay in seconds before retrying failed asset logs configuration parameter before trying again.

Maximum delay in seconds before retrying failed asset logs

The Maximum delay in seconds before retrying failed asset logs configuration parameter contains the maximum value for asset log transmission retry interval. In other words, the first time Scrip 22 cannot reach the ASI server, the number of seconds it will wait before retryuing to send an asset log will not be greater than that specified in the Maximum delay in seconds before retrying failed asset logs configuration parameter.



The defaults values for the Initial delay in seconds before retrying failed asset logs and the Maximum delay in seconds before retrying failed asset logs configuration parameters are respectively 60 and 3600. This means that if the ASI client does no succeed in sending the asset logs to the ASI server the first time, it will retry sending them at intervals (in seconds): 60, 120, 240, , 300, 300, 300 ...



If the ASI client can reach the ASI server, but the ASI server returns an error saying that it is too busy, Scrip 22 will use the Initial limit on random backoff in seconds and the Maximum limit on random backoff in seconds configuration parameters described below to manage the asset log transmission process.

In this case, the ASI client is competing with other ASI clients to reach the ASI server, and it has no way of knowing how many others are involved, and how long they are taking. This is very similar to the situation that happens when there is a collision on a coaxial Ethernet network. So, we address it the same way, with a random binary backoff. This is just like a fixed binary backoff, except that the value being backed off represents the MAXIMUM delay, and the actual delay is generated randomly as a number between zero and the maximum delay.

Initial limit on random backoff in seconds

The Initial limit on random backoff in seconds configuration parameter contains the starting value for the maximum retry interval. In other words, the first time Scrip 22 gets a server busy response, it will wait somewhere between zero and the number of seconds entered in the Initial limit on random backoff in seconds configuration parameter before trying again.

Maximum limit on random backoff in seconds

The Maximum limit on random backoff in seconds configuration parameter contains the upper limit for the maximum retry interval. In other words, if Scrip 22 continues to get server busy responses, it will never wait longer than the number of seconds entered in the Maximum limit on random backoff in seconds configuration parameter before trying again, and it will end up waiting somewhere between zero and this many seconds between retries.







Please note that whenever Scrip 22 succeeds in sending a log to the ASI server, it will reset the Initial limit on random backoff in seconds and the Maximum limit on random backoff in seconds configuration parameters back to their default values.



Maximum number of asset logs to queue

In the Maximum number of asset logs to queue configuration parameter you enter the maximum number of Scrip 61 (System Survey) logs that the client will store on each local system.

The asset log queue is circular. When it reaches the maximum length, as defined by the value of the Maximum number of asset logs to queue configuration parameter, the next log added to the queue replaces the oldest log in the queue.

Send asset logs now

Clicking on the Send asset logs now button will immediately send all the asset logs that are in the message queue to the ASI server for the local site.
