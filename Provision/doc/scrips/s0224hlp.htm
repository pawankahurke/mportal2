Scrip 224 – MS Windows Service Manager

Overview

Automated Support Infrastructure (ASI) service management is the first and only remote service management facility that lets users manage MS Windows services on any number of systems, simultaneously, regardless of their location.

The ASI service management function duplicates the Microsoft Windows service management function. It lets you start, stop, re-start, enable, disable, and change service type from manual to automatic, and vice versa.

ASI service management includes two major functions not supported by the Microsoft Windows service management function:

  • You can manage services on multiple systems at the same time, remotely, regardless of their location. You can perform any action on one or more services on any number of systems simultaneously.

  • You can configure a service to always be kept running, or never be allowed to run.

In addition to being a powerful system management tool, the ASI service management function provides you with comprehensive information on all services defined on a system, not just those enabled or disabled, and updates this information in real-time.

Enabling the ASI service management function

You take all MS Windows service related actions on the ASI server via the MS Windows service console accessible via the AS dashboard. Scrip 224 (MS Windows Service Manager) performs those actions on the one or more systems you took them on.

Before you can start managing MS Windows services on systems at your sites, you need to enable Scrip 224 (MS Windows Service Manager) on the systems where you want to manage MS Windows services.

Scrip 224 can run in polling or event driven mode.

In polling mode, the Scrip runs at a scheduled time at which he surveys MS Windows services on a system, and reports all MS Windows service information to the ASI server.

In event-driven mode, Scrip execution is triggered in real-time by any MS Windows service change.

Please note that when running in event-driven mode, Scrip 224 will tend to run more frequently, accessing the ASI server more frequently.

The table below describes the information contained in the logs produced by this Scrip.

Client time:

Date and time at which event was detected by the Client on the end-user system (e.g. 2001-03-02 19:21:36)

Server time:

Date and time at which event log was received and stored in the SQL database on the Web log server (e.g. 2001-03-02 19:21:53)

Scrip number:

Number of Scrip (no particular relevance)

Customer:

Name of supported customer, individual department or site

Machine:

Name of system as assigned at the time of Windows configuration

Username:

Name of user when event occurred

Client version:

Client version number (e.g. 0.700.2048.LE)

Client size:

Size of Client in bytes

Priority:

Priority of event ranging between 1 (lowest) and 9 (highest)

Description:

Description of event that took place (e.g. symptom detected, problem resolved or automated procedure executed)
