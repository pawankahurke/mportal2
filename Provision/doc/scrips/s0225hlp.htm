Scrip 225 – Directory and File Change Detection and Prevention

Overview

Scrip 225 is a powerful intrusion protection tool that also automatically performs a useful problem resolution function.

It lets you monitor any file and/or directory on one, some, or all systems at a site, detect changes, and prevent them if configured to do so.

Scrip 225 operation

You can configure Scrip 225 to monitor files and/or directories continuously, in which case the Scrip will detect changes to directories and files as they happen, or on a schedule, in which case it will check them at the time you define in the Scrip’s configuration parameter.

If you enable Scrip 225’s change prevention function, the Scrip will make a copy of the files that you want to prevent from being changed, and store all the other relevant information. If you choose to protect a directory, it will copy the entire directory, and all its other relevant information.

Scrip 225 runs when one of the following events occurs:

  • When the ASI client starts up. When this happens it checks for changes

  • When you click on the execute button to the right of the Check monitored items now label in the Scrip’s configuration page. When this happens it checks for changes.

  • At its scheduled time. When this happens it checks for changes

  • When monitored item(s) change. When this happens it checks for changes.

  • When one of its configuration parameter changes. When this happens it updates the configuration.

  • When it is enabled/disabled. When this happens it updates the configuration.

Checking for changes means:

  • Determining if any files or directories have been removed, added or altered, and

  • Updating the stored information about any item that has changed

Updating the configuration means updating the stored information about monitored item(s) to match their current state. If any existing stored information for monitored item(s) is still relevant, then it is re-used. If a new item is added, it means recording its information.

If Scrip 225 is configured to prevent changes, instead of updating the stored information about any item that has changed, it will restore the item to its pre-change state by overwriting the changed item with the copy that was stored when the item was first added to the Monitored items parameter as (part of) a command line.

This means that if a protected item is corrupted because of a system’s software or hardware malfunction, and does not function properly any longer, if the ASI client is operational, Scrip 225 will restore the protected item to its working state. In this scenario,, the Scrip can be a useful automated problem resolution tool.

The table below describes the information contained in the logs produced by this Scrip.

Client time:

Date and time at which event was detected by the Client on the end-user system (e.g. 2001-03-02 19:21:36)

Server time:

Date and time at which event log was received and stored in the SQL database on the Web log server (e.g. 2001-03-02 19:21:53)

Scrip number:

Number of Scrip (no particular relevance)

Customer:

Name of supported customer, individual department or site

Machine:

Name of system as assigned at the time of Windows configuration

Username:

Name of user when event occurred
